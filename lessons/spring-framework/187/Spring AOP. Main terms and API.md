# Spring AOP. Основные термины и API. Немного об AspectJ

Сегодня углубимся в теоретическую составляющую AOP - рассмотрим основные термины, которыми приходится оперировать в 
данной парадигме. Также рассмотрим привязку этих терминов к API, который предоставляет AspectJ. И которыми, 
соответственно, оперирует Spring AOP.

## Join point

**Join point** - точка выполнения программы, в которую внедряется аспект. В общем случае это может быть практически 
любой вызов в байт-коде: вызов метода, создание объекта, получение доступа к полю, выбор исключения и так далее. В 
случае со Spring AOP все намного проще - join point всегда будет привязан строго к вызову метода. Это фундаментальное
ограничение, обусловленное привязкой Spring AOP к прокси.

В AspectJ join point находит отражение в интерфейсе `JoinPoint` и его наследниках. При этом если концептуально join 
point обозначает некое место в коде (байт-коде), к которому привязывается аспект, то использование `JoinPoint` в 
кодовой базе фактически оперирует вызовом внутри advice'а (эдвайса) и будет обозначать вызов целевого метода (метода 
target object'а) или следующего advice'а в цепочке.

И термин advice, и термин target object уже использовались ранее, но более подробно разбираются ниже.

Сам по себе `JoinPoint` предоставляет методы информационного характера. Мы не будем разбирать все из них подробно, 
однако тезисно рассмотрим их назначение:

- `toString()`, `toShortString()`, `toLongString()`. Данные методы предоставляют строковую интерпретацию выражения, 
  которое описывает вызов текущего advice'а. Скажем, если advice вызывается на основе AspectJ-выражения
  `execution(* com.walking.*.*(..))`, вызвав указанные методы внутри эдвайса мы будем видеть более узкое выражение, 
  объясняющая непосредственно текущий вызов. То есть для вызова, сработавшего для `MyComponent1#test1()` мы увидим 
  строку вида `execution(void com.walking.springsample.MyComponent1.test1())`, для `MyComponent2#test2()` - 
  `execution(void com.walking.springsample.MyComponent2.test2())`. Между собой данные методы `JoinPoint` отличаются 
  в первую очередь форматом вывода и его подробностью. В первую очередь они полезны для логирования вызовов эдвайсов 
  и отладки;
- `getThis()`. Возвращает реальный объект, для которого сработал эдвайс. Поскольку Spring AOP работает на базе 
  прокси - это будет либо объект CGLIB-прокси (скажем, `MyComponent1$$SpringCGLIB$$0`), либо объект JDK dynamic proxy 
  (`Proxy100500`). При более глубокой интеграции AspectJ в приложение результат может быть иным, это чуть 
  подробнее описано в пункте, посвященном weaving;
- `getTarget()`. Возвращает target object - тот объект, для которого задумывалось применение эдвайса изначально. В 
  нашем примере это может быть объект типа `MyComponent1`. Как правило, в целях логирования или интроспекции нас 
  будет интересовать именно target object и почти никогда - результат `getThis()`. Последний полезен в основном для 
  отладки при каких-либо проблемах, возникших в работе со Spring AOP;
- `getArgs()`. Возвращает параметры метода, с которыми был вызван эдвайс. В случае, если параметры были подменены 
  эдвайсом, сработавшим ранее - поведение может быть не очевидным. Следующие эдвайсы в цепочке и сам целевой метод 
  будут получать уже подмененные параметры, но в пределах эдвайса, подменяющего их, `JoinPoint` все так же будет 
  предоставлять те значения, которые были на момент вызова эдвайса;
- `getSignature()`. Предоставляет метаданные join point в виде объекта `Signature`. В упрощенном виде можно 
  рассматривать как обертку над Reflection API для получения доступа к информации о целевом методе. Как и предыдущие 
  методы, полезен в основном для логирования и отладки;
- `getSourceLocation()`, `getKind()`, `getStaticPart()`. В Spring AOP с runtime weaving (см. соответствующий
  пункт) данные методы фактически бесполезны. Детали работы при глубокой интеграции AspectJ находятся за пределами 
  курса и, в целом, не представляют интереса, пока данный подход не потребовался на практике.

Именно объектом `JoinPoint` оперирует разработчик в большинстве эдвайсов. Такой объект не позволяет как-либо влиять 
на целевой вызов. Влияние на вызов предполагается только при использовании `@Around`-эдвайсов, ведь в них необходимо 
явным образом указать, в какой момент управление должно быть передано целевому методу (или следующему эдвайсу). И 
для таких случаев используется `ProceedingJoinPoint`.

`ProceedingJoinPoint` является наследником `JoinPoint` и предлагает несколько методов, связанных с вызовом целевого 
объекта. Для использования в эдвайсах предполагается только метод `proceed()` в нескольких вариациях. Он фактически 
передает управление дальше по цепочке, именно через него появляется возможность подменить параметры вызова, получить 
результат этого вызова - возвращаемое значение или исключение (требуется `catch` или `throws`-блок).

Основные особенности вызова `ProceedingJoinPoint#proceed()` были описаны в предыдущей статье.

## Advice

Advice - логика внутри аспекта, применяемая к конкретному целевому объекту. Аспект может содержать несколько 
эдвайсов, которые будут иметь общие или различные критерии применения. О самих критериях подробнее рассказано в 
пункте "Pointcut". 

В Spring AOP эдвайс всегда будет выглядеть как метод, помеченный одной из аннотаций:

- `@Around`. Наиболее широкая по зоне ответственности аннотация из всех, определяющих метод как advice. Позволяет 
  совершать какие-то действия как до, так и после целевого метода. Технически позволяет подменять как сам целевой 
  метод (внутри эдвайса нет вызова `ProceedingJoinPoint#proceed()`), так и его параметры (переопределение атрибутов 
  через `proceed()`) и результат. `@Around`-эдвайсы в норме описываются как методы, возвращающие `Object`, иные 
  ситуации приведут к потере возвращаемого значения целевого метода. Также, если нет цели отлавливать ошибки, возникшие
  при обработке целевого метода, эдвайс снабжается `throws`-блоком. Именно через работу с возвращаемым методом эдвайса и
  исключениями можно подменять результаты работы целевого метода. Стоит ли так делать - вопрос дискуссионный, но чаще
  всего правильным ответом будет "нет";
- `@Before`. Аннотация эдвайса, позволяющая выполнять определенную логику строго _до_ вызова целевого метода;
- `@After`. Аннотация эдвайса, позволяющая выполнять определенную логику строго _после_ вызова целевого метода. При 
  этом неважно, завершилась ли работа целевого метод корректно или с исключением. Условно, можно 
  представить, что целевой метод вызывается внутри `try`, а `@After`-эдвайсы - в `finally`-блоке;
- `@AfterReturning`. Аннотация эдвайса, позволяющая выполнять определенную логику строго _после успешного_ вызова 
  целевого метода. Второе отличие от `@After` - возможность получить доступ к возвращаемому значению метода с 
  помощью атрибута `returning`. Пример приведен ниже;
- `@AfterThrowing`. Аннотация эдвайса, позволяющая выполнять определенную логику строго после вызова целевого метода,
  завершившегося ошибкой. Также, как `@AfterReturning` позволяет работать с возвращаемым значением метода, 
  `@AfterThrowing` имеет возможность настроить доступ к выброшенному исключению через атрибут `throwing`.

Достаточно очевидно, что любая из аннотаций выше может быть заменена на `@Around`. Фактически, остальные аннотации - 
не более чем синтаксический сахар, избавляющий от необходимости работать с `ProceedingJoinPoint` и позволяющий 
быстрее оценить зону применения конкретного эдвайса при чтении кода.

На мой взгляд, рассмотренный выше API весьма дружелюбен и интуитивно-понятен. Непокрытыми остается лишь один вопрос:
какие сигнатуры допустимы для эдвайс-методов? Ведь в зависимости от тех или иных параметров мы будем получать внутри
эдвайса доступ к той или иной информации.

### Сигнатура эдвайса

Формально в понятие сигнатуры не включается тип возвращаемого значения, но начать стоит именно с него.

Для `@Around`-эдвайсов возвращаемое значение метода в норме проксирует возвращаемое значение целевого метода. Таким 
образом, если мы не хотим это значение потерять или заменить - эдвайсу имеет смысл возвращать `Object` или более 
узкий тип (если мы уверены, что этот тип будет совместим с возвращаемым значением целевого метода). Во всех 
остальных случаях возвращаемое значение эдвайса никак не используется, соответственно, `void` будет наиболее 
адекватным выбором.

Название метода-эдвайса может быть любым - нет каких-то общих правил именования. Чаще всего логика будет той же, что 
и у обычного метода - некая мнемоника, минимально описывающая назначение метода. Аналогично и со `throws`-блоком - 
чаще всего его наличие будет обуславливаться логикой самого эдвайса. Единственное строгое правило - если
`@Around`-эдвайс не направлен на обработку исключений, стоит пробрасывать `Throwable`. Но поскольку это часть 
сигнатуры `ProceedingJoinPoint#proceed()`, неумышленно обойти такое ограничение проблематично.

Самое интересное - параметры метода-эдвайса. Все из описанных ниже параметров формально необязательны, но на практике 
за каждой из рассмотренных выше аннотаций будет достаточно жестко следовать тот или иной набор параметров.

#### JoinPoint

`ProceedingJoinPoint` доступен только для `@Around`, все остальные могут оперировать `JoinPoint`. Имя параметра в 
данном случае не имеет значения, важен только тип.

#### Параметры целевого метода

В общем случае параметры target-метода и их значения доступы через `JoinPoint`. В некоторых случаях для удобства они 
могут быть вынесены в параметры самого эдвайса, но это требует определенных действий при настройке эдвайса:

1. Добавление нужных параметров в сигнатуру эдвайс-метода;
2. Добавление имен параметров в аргумент эдвайс-аннотации `argNames`. Такой атрибут присутствует и в `@Around`, и во 
   всех остальных аннотациях, рассмотренных выше;
3. Добавление проверки наличия параметра в pointcut - выражение, определяющее, в каких случаях срабатывает эдвайс. 
   Это те самые AspectJ-выражения, которые упоминались в предыдущей статье и при знакомстве с API `JoinPoint` выше. 
   Если подобной проверки не будет - эдвайс может посчитаться невалидным и просто игнорироваться.

На практике это будет выглядеть примерно так:

```java
@Component
public class MyComponent {
    // myArg - тот параметр, который мы попытаемся передавать в метод эдвайса
    @MyAnnotation
    public void test(int myArg) {
        System.out.println("Test method call!");
    }
}

@Aspect
@Component
public class MyAspect {
    // args(myArg) - Эдвайс будет применен только к тем методам, у которых есть параметр с именем myArg. 
    // Соответственно, pointcut в данном случае проверяет сразу два условия.
    // argNames = "myArg" - указывает, что мы хотит привязать параметры целевого метода с параметрам эдвайса. Если 
    // потребовалось привязывать несколько параметров - в argNames они бы указывались единой строкой через запятую.
    @Around(value = "@annotation(MyAnnotation) && args(myArg)", argNames = "myArg")
    // myArg будет содержать значение параметра, указанное при вызове целевого метода. Соответствие имен в 
    // методе-эдвайса и аннотации выше - обязательно, т.к. это основа привязки. Тип парметра result может быть любым,
    // но необходимо быть уверенным в совместимости типа параметра эдвайс-метода и параметра целевого метода
    public void process(ProceedingJoinPoint joinPoint, int myArg) throws Throwable {
        // ... - Логика эдвайса
    }
}
```

По сути, ничего супер-сложного нет, но в большинстве случаев гораздо проще работать с параметрами целевого метода через
`JoinPoint`.

#### Аннотация

Если в pointcut учитывается аннотация над целевым методом (часто использовалось в примерах предыдущей статьи) или 
классом - эту аннотацию можно передавать как параметр метода-эдвайса. Пример также есть в предыдущей статье. Это 
полезно в случаях, когда необходимо узнать атрибуты аннотации именно для целевого типа или метода.

#### Результат выполнения метода

Для `@AfterReturning` существует возможность передать возвращаемое значение целевого метода в параметр эдвайс-метода.
Это позволяет, например, модифицировать возвращаемое значение (если оно представлено мьютабельным объектом). 
Механика чем-то напоминает работу с `argNames`:

```java
@Component
public class MyComponent {
    @MyAnnotation
    public int test() {
        return 1;
    }
}

@Aspect
@Component
public class MyAspect {
    // returning = "result" обеспечивает привязку возвращаемого значения целевого метода к параметру "result" 
    // метода-эдвайса. Имя параметра может быть любым, главное, чтобы оно совпадало со значением в
    // @AfterReturning#returning.
    @AfterReturning(value = "@annotation(MyAnnotation)", returning = "result")
    // Тип параметра result может быть любым. Наиболее общим решением будет Object. С любым другим типом необходимо 
    // обладать уверенностью, что тип параметра совместим с типом возвращаемого значения, иначе при вызове произойдет 
    // ошибка приведения типа 
    public void process(JoinPoint joinPoint, Object result) throws Throwable {
        // ... - Логика эдвайса
    }
}
```

Для `@AfterThrowing#throwing` логика привязки будет аналогичной, но для выброшенного целевым методом исключения.

## Pointcut

**Pointcut** - выражение, определяющее набор join point'ов, к которым должен примениться advice. Во всех примерах 
этой и предыдущей статей pointcut'ами являются выражения вроде `execution(* com.walking.*.*(..))` или `@annotation
(MyAnnotation)`.

В Spring AOP существует два способа определить pointcut:

1. Описать его в аннотации эдвайса через атрибут `value` или `pointcut`. Последнее есть только у `@AfterReturning` и 
   `@AfterThrowing`, являясь семантическим аналогом `value`, но с большим приоритетом - если заполнены оба, будет 
   использоваться именно `pointcut`;
2. Создать именованный pointcut - пустой метод, помеченный аннотацией `@Pointcut` с описанным в ней выражением. 
   Данный способ удобен, если в аспекте присутствует несколько эдвайсов с одинаковым поинткатом.

В обоих случаях описание выражения происходит по правилам AspectJ expression. На базовом уровне мы с этим языком уже 
знакомы, более подробно углубимся в следующей статье.

Как и в остальных частях Spring AOP, API описания поинткатов выглядит дружелюбным, остается лишь добавить пример.

Знакомый нам формат

```java
@Aspect
@Component
public class MyAspect {
    @Before("@annotation(MyAnnotation)")
    public void processAspect(JoinPoint joinPoint) {
      // ... - Логика эдвайса
    }
}
```

Может быть оформлен как

```java
@Aspect
@Component
public class MyAspect {

    @Pointcut("@annotation(MyAnnotation)")
    public void myPoincut() {}
    
    @Before("myPoincut()")
    public void processAspect(JoinPoint joinPoint) {
      // ... - Логика эдвайса
    }
}
```

Также существует возможность привязки параметров целевого метода в поинткате, по аналогии с тем, как мы это делали 
для эдвайс-аннотаций:

```java
@Aspect
@Component
public class MyAspect {
    @Around(value = "@annotation(MyAnnotation) && args(myArg)", argNames = "myArg")
    public void process(ProceedingJoinPoint joinPoint, Object myArg) throws Throwable {
        // ... - Логика эдвайса
    }
}
```

Можно превратить в

```java
@Aspect
@Component
public class MyAspect {
    @Pointcut(value = "@annotation(MyAnnotation) && args(myArg)", argNames = "myArg")
    public void myPointcut(Object myArg) {}

    @Around(value = "myPointcut(myArg)", argNames = "myArg")
    public void process(ProceedingJoinPoint joinPoint, Object myArg) throws Throwable {
      // ... - Логика эдвайса
    }
}
```

Что обеспечит переиспользуемость поинтката и упростит его изменение.

## Target object

Предположу, что к данному момент термин **target object** уже прояснился - это фактический объект, к которому будут 
применены advice'ы, он находится "под" всеми накрученными на него прокси-типами.

В этой концепции нет ничего сложного, для пользовательских бинов таким объектом чаще всего будет либо объект, созданный
из компонента, либо результат выполнения `@Bean`-метода.

Однако в этом пункте также стоит дать несколько вспомогательных инструментов, полезных для отладки, погружения в 
реализацию и, в некоторых случаях, для тонкой настройки логики эдвайсов:

- `AopProxyUtils`. Утилитный класс, который предоставляет методы для получения деталей текущего прокси-объекта, в 
  том числе - получение целевого объекта или его класса;
- `AopUtils`. Еще один утилитный класс, позволяющий убедиться, что рассматриваемый объект - прокси, созданный Spring 
  AOP. Также позволяет узнать используемый инструмент проксирования (JDK dynamic proxy или CGLIB), проверить 
  соответствие объекта поинткату и некоторые другие вещи.

В прикладном коде данные классы встречаются редко, но для песочницы на этапе погружения в Spring AOP бывают довольно 
полезны, особенно для узких сценариев, которые сложно нагуглить, и экспериментов.

## Weaving

Последний термин, который необходимо разобрать сегодня - **weaving**.

Weaving - механизм встраивания сквозной логики, поставляемой аспектами, в целевые объекты. Иными словами, это именно 
тот движок, который позволяет Spring AOP или иному подобному инструменту применять эдвайсы в нужный момент.

На данном этапе нам достаточно поверхностно разобраться с основными подходами к реализации weaving'а:

- Runtime weaving;
- Compile-time weaving;
- load-time weaving.

### Runtime weaving

Это подход к встраиванию аспектов на этапе инициализации `ApplicationContext`. Именно этот подход используется в Spring 
AOP по умолчанию.

Фактически он сводится к тому, что для работы с аспектами у бинов формируются прокси-классы, в логику которых и 
включается применение эдвайсов.

> Именно реализацию Runtime weaving в Spring AOP также иногда называют proxy-based. Это более точно описывает 
> используемый подход.

Основные минусы этого подхода:

- AOP доступен только в пределах Spring-бинов. Для Spring AOP как такового это не проблема, для реализации АОП как 
  самостоятельной концепции - очевидно, да;
- Естественным образом наследует ограничения проксирования - их разбирали в предыдущих статьях.

К плюсам этого подхода можно отнести:

- Простой процесс сборки приложения. Нет необходимо добавлять кастомные компиляторы, Java-агенты и другие 
  относительно низкоуровневые возможности JVM;
- В пределах Spring-контекста все работает бесшовно. Неважно, как разбита кодовая база приложения - на модули, 
  библиотеки или иные структурные блоки. Для любых Spring-бинов АОП будет работать предсказуемо;
- Поскольку это решение по умолчанию, именно для него существует наиболее широкая база знаний в Интернете в целом и 
  в профильных сообществах в частности.

### Compile-time weaving (CTW)

Основная альтернатива - weaving на этапе компиляции. Это подход, который по умолчанию использует "ванильный" AspectJ.

Суть механизма заключается в связывании целевого кода с аспектами еще на этапе компиляции - в сам байт-код классов 
приложения будут встроены вызовы подходящих им эдвайсов.

Основные плюсы:

- Легкий старт приложения. Weaving произошел на этапе компиляции, на него не тратятся ресурсы ни при запуске 
  приложения, ни в ходе его работы;
- Возможности кратно шире. Можно применять аспекты к приватным методам, нет проблем с применением аспектов при 
  self-invocation и т.д. Нет прокси - нет связанных с ними проблем;
- Работает не только со Spring-бинами, но и вообще с любыми классами приложения.

Минусы:

- Spring обеспечивает поддержку CTW, но она не включается автоматически. То есть мы можем подключить в 
  Spring-приложение AspectJ, полностью его настроить и пользоваться всеми его возможностями. Все будет работать до тех
  пор, пока мы не захотим внедрить какой-нибудь Spring-бин в аспект, определить порядок выполнения эдвайсов через
  механизмы Spring или иным образом связать аспекты с возможностями Spring Framework. Такую связь тоже можно 
  обеспечить, но это требует дополнительной конфигурации приложения. При этом какие-то стандартные возможности Spring 
  Framework могут быть переведены на CTW без особых проблем, но где-то придется помучиться с внедрением;
- Применение аспектов к библиотекам или отдельно компилируемым модулям. Аспекты модуля могут встроиться в join 
  point'ы в пределах этого модуля. Но если что-то уже поставляется в скомпилированном виде - увы. AspectJ 
  предоставляет механизмы решения этой проблемы через переработку существующего байткода, однако это сильно сложнее, 
  чем привычный нам процесс сборки приложения.

Технически, довольно много задач, решаемых через АОП не требуют использования Spring. Остальные можно решить, 
углубившись в настройку и добавив в проект несколько зависимостей. Проблемы доступности аспектов для уже 
скомпилированных модулей/библиотек тоже можно решить. Но по совокупности почти никто не готов жертвовать привычками и
удобством стандартной интеграции ради чуть более быстрого запуска и работы системы. Вероятно, поэтому AspectJ в 
чистом виде в принципе малопопулярен.

### Load-time weaving (LTW)

Load-time weaving - вероятно, наиболее продвинутая и, одновременно, наиболее технически-сложная реализация 
weaving-механизма. Она предлагает обеспечивать привязку advice'ов к join point'ами в процессе загрузки классов class 
loader'ом, изменяя уже существующий байт-код.

Этот подход существует, но не является решением по умолчанию ни для Spring AOP, ни для AspectJ. Как и compile-time 
weaving, он требует предварительной дополнительной настройки, в том числе на этапе запуска приложения.

В остальном Load-time weaving может быть даже удобнее, нежели CTW. С одной стороны, подготовка проекта и окружения 
чуть более муторная, чем с CTW, с другой - пропадает ряд нюансов вроде необходимости учитывать доступность АОП в 
каждой отдельно взятой подключаемой библиотеке. Если скорость запуска приложения не критична - LTW кажется более 
прозрачным подходом, нежели compile-time weaving. 

Но, как и с любыми непопулярными решениями стоит отдавать себе отчет, что с любыми нетиповыми проблемами скорее всего 
придется разбираться самостоятельно и ключей к решению не будет ни на первой, ни на пятой странице в поисковике:)

> На мой взгляд, junior-специалисту в принципе не обязательно знать слово weaving. Однако в текущих реалиях стоит уметь
> хотя бы в двух предложениях описывать основные концепции в технологиях, о которых спрашивают на собеседованиях.
>
