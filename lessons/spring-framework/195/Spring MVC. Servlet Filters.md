# Spring MVC. Работа с сервлетными фильтрами

Сегодня постараемся разобраться с особенностями использования сервлетных фильтров в Spring MVC-приложениях: сценарии
использования, регистрация, доступ к IoC-контейнеру внутри фильтра и так далее.

## Сценарии использования

Глобально мы уже знакомы как с концепцией сквозной функциональности, так и с типовыми случаями для использования 
сервлетных фильтров. Дублировать эту информацию здесь нет необходимости.

Основной момент, который стоит разобрать - сравнение сервлетных фильтров и `HandlerInterceptor`, с которым мы 
познакомились в одной из предыдущих статей.

В принципе, и интерцепторы в Spring MVC, и фильтры предназначены для примерно одних и тех же целей и в некоторых 
случаях неважно, какой из инструментов будет выбран для конкретной задачи. Однако существует ряд как технических, 
так и идеологических нюансов, исходя из которых либо фильтр, либо интерцептор будет предпочтителен в конкретной
ситуации. В целом, такие нюансы следуют из понимания общего процесса обработки запроса с помощью Servlet API и 
Spring MVC, но для наглядности отметим их в данном пункте.

### Обертки запроса и ответа

Одна из наиболее частых возможностей, используемых в сервлетных фильтрах - подмена оригинального объекта запроса или 
ответа соответствующим объектом-враппером. Причин создания такой обертки может быть много, наиболее частая из них - 
возможность несколько раз читать тело запроса или ответа. Это, в свою очередь, полезно для кэширования, валидации 
подписи (например, HMAC) и так далее.

Эта возможность прямо следует из того, что фильтр сам определяет момент и параметры вызова следующего 
фильтра в цепочке. Соответственно, подобный механизм функционально невозможен в перехватчиках.

Нужна обертка запроса или ответа - однозначно должен использоваться фильтр.

### Работа с обработчиком запроса

Несложно заметить, что все методы `HandlerInterceptor` содержат параметр `Object handler`. В случае с классическими 
аннотированными контроллерами речь всегда будет идти об объекте `HandlerMethod`, содержащем объект бина 
соответствующего контроллера, объект метода этого контроллера и ряд другой информации.

Кроме того, механизм регистрации `HandlerInterceptor` позволяет их донастройку - например, ограничение URL-паттернов,
по которым конкретный interceptor должен (или не должен) быть применен. Детали можно рассмотреть в рамках API 
`WebMvcConfigurer`. Конкретно в данном случае нас будет интересовать метод `addInterceptors()`.

Стоит отметить, что ранее в курсе мы не затрагивали `WebMvcConfigurer`. Работа с ним будет рассмотрена в одной из 
последних статей в блоке Spring MVC.

Фильтр, на момент начала своей работы, вообще ничего не знает и не может знать ни о контроллерах как сущности, ни о 
том, каким конкретно методом контроллера будет обработан конкретный запрос и будет ли вообще (`HandlerMapping` тоже еще 
неизвестен на момент вызова фильтра). Конечно, после того, как фильтр снова получает управление (после отработки 
запроса) такую информацию уже можно попробовать извлечь, но на общую концепцию это не влияет.

Соответственно, вся функциональность, касающаяся работы с классом или методом контроллера, обработкой 
аннотаций соответствующих сущностей и так далее - ответственность `HandlerInterceptor`.

В более широком смысле к ответственности `HandlerInterceptor` иногда относят вообще любую сквозную функциональность, 
так или иначе связанную с механизмами Spring. Фильтры в таком подходе отвечают только за специфику Servlet API и 
различные инфраструктурные нюансы. Однако такая позиция, на мой взгляд, является слишком категоричной.

### Порядок обработки

Достаточно узкий пункт, который, может оказаться решающим в разрезе отдельных задач.

Из общего порядка действий при обработке запроса в Spring MVC очевидно, что:

1. Логика сервлетного фильтра, описанная до передачи фильтром управления дальше по цепочке, выполнится как до 
   следующих фильтров цепочки, так и до любых методов `HandlerInterceptor`;
2. Методы `HandlerInterceptor`, вызываемые после выполнения метода контроллера (`postHandle()`, `afterCompletion()`) 
   выполняются до логики сервлетного фильтра, описанной после передачи фильтром управления дальше по цепочке*.

> *В порядке вызова есть некоторые нюансы, если включена асинхронная обработка запросов. Но на данном этапе это 
> можно вынести за скобки - на саму методологию выбора между фильтром и интерцептором это обычно не влияет.

Соответственно, если логика, которую необходимо реализовать в фильтре или перехватчике хотя бы косвенно зависит от 
другого класса со сквозной функциональностью, необходимо определять, в какую точку общего процесса ее встраивать. 
Это касается как приоритета отдельных фильтров и перехватчиков между собой, так и позиционирования друг относительно 
друга.

Сюда, например, относится обработка ошибок. Важно понимать, какие ошибки в конкретном приложении обрабатываются 
механизмами Spring (`@ExceptionHandler`-методы и `HandlerExceptionResolver` в принципе), а какие могут быть 
выброшены за пределы `DispatcherServlet` и отловлены в фильтре. На практике последнее - редкий сценарий использования,
но имеющий право на жизнь.

Но чаще всего на этот пункт смотрят в разрезе безопасности. Гораздо шире данный вопрос мы будем разбирать в разрезе 
Spring Security, но общая идея проста: чем быстрее сервер определит невалидный запрос, тем меньше ресурсов он 
потратит на его обработку.

А раз фильтр отрабатывает до перехватчика - он явно предпочтительный кандидат. Сюда относится валидация авторизации, 
CORS-политик, CSRF-защита и многие другие вещи, которые не требуют понимания бизнес-смысла конкретного HTTP-запроса.
Базовые механизмы обычно покрываются стандартными конфигурациями Spring Security, но отдельные специфичные вещи иногда 
приходится делать вручную.

### Гарантия вызова

По сути, данный пункт снова возвращает нас к особенности обработки ошибок. При прочих равных, фильтр будет 
обрабатывать больше запросов, нежели `HandlerInterceptor`. И если требуемая функциональность должна срабатывать в 
негативных сценариях - фильтр часто окажется предпочтительным вариантом для размещения такой функциональности. 
Несколько примеров для наглядности:

- Обработчика не существует - запрос пришел по неверному пути или по какой-то еще причине не нашелся `HandlerMapping`
  для него. `HandlerInterceptor` не отработает вовсе, поскольку привязан к конкретному `HandlerExecutionChain`, а 
  его-то и не удалось найти;
- Ошибка при обработке логики контроллера. `HandlerInterceptor#postHandle()` не сработает, лишь `afterCompletion()`.

### Доступность Spring-бинов в логике

Технически, и фильтр, и перехватчик может как быть, так и не быть Spring-бином. Соответственно, для каждого из них 
может как быть, так и не быть доступен DI и другие механизмы Spring.

При этом корректно зарегистрировать интерцептор в виде бина чуть проще, чем фильтр, но это технический нюанс. В 
разрезе выбора того или иного механизма это лишь дополнительный маркер: если вам потребовались Spring-бины в 
сквозной логике - чаще (но не всегда) это говорит о том, что стоит использовать именно интерцептор, а не фильтр.

Однако это все еще второстепенный маркер, основным критерием должна выступать сама функциональность и требования, 
предъявляемые к ней. Иными словами, тезисы выше имеют гораздо большее значение, чем текущий.  

### Логика, оборачивающая обработку запроса

Еще один стилистический момент. Сам Java API фильтров говорит, что сделать что-то вроде:

```java
protected void doFilter(HttpServletRequest req, HttpServletResponse res, FilterChain chain)
        throws IOException, ServletException {
    var preHandleResult = doSth();
    
    chain.doFilter(requestWrapper, responseWrapper);
    
    handleFinalResult(preHandleResult);
}
```

В случае с фильтром проще, чем с `HandlerInterceptor`. Для последнего как минимум придется изобретать на коленке 
механизм передачи данных, полученных в `preHandle()` в один из методов, пост-обработки.

Соответственно, если сквозная функциональность предполагает что-то, что в Spring AOP advice мы бы описывали через 
`@Around`-метод - чаще всего стоит описать функциональность в виде фильтра. Опять же, это лишь техническое 
ограничение, которое можно преодолеть и в интерцепторе, но чаще именно фильтр окажется верным выбором для подобных 
ситуаций.

## Регистрация фильтра

На этом теоретическая составляющая завершена, осталось рассмотреть несколько прикладных нюансов.

Первый из них - нам нужно как-то сообщить приложению, что существует сервлетный фильтр, который должен применяться, 
при этом также нужно иметь возможность как-то указывать порядок применения фильтров.

В классическом сервлетном приложении мы бы реализовали и регистрацию фильтра, и описание его маппинга через `web.xml`,
но в Spring MVC вариаций больше:

- `web.xml`. Технически, никто не запрещает использовать его в Spring MVC приложениях, но это не слишком популярно;
- `AbstractAnnotationConfigDispatcherServletInitializer`. Мы уже работали с этим классом. Если нам достаточно 
  зарегистрировать фильтр для конкретного `DispatcherServlet` - существует выделенный метод для этого. Пример будет 
  приведен ниже. Учитывая, что чаще всего в современных приложениях лишь один `DispatcherServlet` - текущий вариант 
  достаточен. Однако стоит понимать, что фактически он привязывает фильтр именно к маппингу `DispatcherServlet`. То 
  есть фильтр будет работать только для путей, указанных в `getServletMappings()`;
- `WebApplicationInitializer`. Данный интерфейс Spring MVC является программной альтернативой `web.xml`. В своем 
  единственном методе `onStartup()` интерфейс предоставляет доступ к `ServletContext`. Что, в свою очередь, 
  позволяет программно регистрировать фильтры через любой из методов `ServletContext#addFilter()`. Соответственно, 
  такой подход никак не привязывает разработчика к конкретному объекту `DispatcherServlet`. Если в приложении 
  несколько `DispatcherServlet` - это имеет смысл. Кроме того, данный подход дает чуть большую гибкость в 
  конфигурации через доступ к `FilterRegistration.Dynamic`.

Также стоит отметить, что в Spring Boot существуют отдельные механизмы добавления фильтров. Их мы рассмотрим в 
отдельном разделе курса. А поскольку в реальном мире вы чаще всего будете сталкиваться именно со Spring Boot 
приложениями, на данном этапе достаточно одного любого способа регистрации фильтров. Поэтому выберем наиболее знакомый:

```java
// Сделаем для наглядности примитивный фильтр. Фактически, он ничего не делает
public class EmptyFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        chain.doFilter(request, response);
    }
}

public class WebAppInitializer extends AbstractAnnotationConfigDispatcherServletInitializer {
    
    // ... - Другие методы, описывающие конфигурацию DispatcherServlet. Детализировать не будем, неоднократно 
    //       рассматривали данный класс в предыдущий статьях
    
    @Override
    protected Filter[] getServletFilters() {
        // Порядок фильтров в возвращаемом массиве определяет и порядок их вызова при обработке запроса 
        return new Filter[]{
                new EmptyFilter()
        };
    }
}
```

## DelegatingFilterProxy

Вторая техническая проблема - использование IoC. По умолчанию фильтр не является бином. Более того, обычно фильтр 
регистрируется по классу, а не по объекту, то есть даже если аннотировать конкретный фильтр бином, это не решит 
проблему.

> Здесь можно вспомнить про WebApplicationInitializer, который предлагает Spring. В принципе, через него доступен
> `ServletContext#addFilter()`, одна из перегрузок которого позволяет передавать объект фильтра, а не класс.
> 
> Проблема здесь заключается в том, как Spring MVC обрабатывает сами реализации `WebApplicationInitializer`. Обычно 
> такие реализации не являются бинами, поскольку их поиск и выполнение - задача `SpringServletContainerInitializer`. 
> `SpringServletContainerInitializer` - реализация сервлетного интерфейса `ServletContainerInitializer` и его 
> обработка - ответственность контейнера сервлетов, а не Spring-приложения.
> 
> Как результат - в базовых сценариях нет возможности корректно зарегистрировать сервлетный фильтр в виде бина. 

Однако сценарии, когда фильтр должен взаимодействовать со Spring-бинами - реальность. И нужен способ, который 
позволит такое взаимодействие реализовать. Им является класс `DelegatingFilterProxy`.

`DelegatingFilterProxy` - класс, реализующий интерфейс `Filter`, что позволяет его использовать как сервлетный 
фильтр. Также он позволяет указать ему бин-делегат через имя бина или сам объект бина. Собственно, основная задача 
данного класса заключается в обеспечении возможности использовать сервлетный фильтр как полноценный Spring bean.

В наиболее классическом сценарии это происходит через регистрацию `DelegatingFilterProxy` с указанием имени класса 
целевого бина. В таком случае, в момент, когда конкретный объект `DelegatingFilterProxy` вызывается контейнером 
сервлетов для обработки запроса, прокси получает бин целевого фильтра по имени и делегирует ему обработку запроса. 
Реализацию несложно понять по исходному коду `DelegatingFilterProxy#doFilter()`.

Регистрация бина при этом может выглядеть примерно так:

```java
// Фильтр должен быть зарегистрирован в контексте. Либо через аннотации вроде @Component, либо через @Bean-метод.
// Имя бина указывать необязательно, но это хороший тон в случае, если мы явно используем имя далее
@Component("springAwareFilter")
public class SpringAwareFilter implements Filter {
    private final MySpringBean mySpringBean;
    
    public SpringAwareFilter(MySpringBean mySpringBean) {
        this.mySpringBean = mySpringBean;
    }
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        mySpringBean.doSth();
        chain.doFilter(request, response);
    }
}

public class WebAppInitializer extends AbstractAnnotationConfigDispatcherServletInitializer {
    
    // ... - Другие методы, описывающие конфигурацию DispatcherServlet
    
    @Override
    protected Filter[] getServletFilters() {
        return new Filter[]{
                // Создаем DelegatingFilterProxy с указанием имени бина, соответствующего целевому фильтру
                new DelegatingFilterProxy("springAwareFilter")
        };
    }
}
```

Для регистрации фильтра через `WebApplicationInitializer` логика использования `DelegatingFilterProxy` будет 
аналогичной.

## OncePerRequestFilter

Третий и последний в данной статье нюанс - особенность работы фильтров при асинхронной обработке запросов.

В самом курсе мы умышленно не затрагивали асинхронную обработку ни на уровне Servlet API, ни в разрезе Spring MVC - 
эта тема не критична для новичков, при этом при ее качественном изучении требует как хорошей базы в смежных темах, в 
том числе близких к архитектуре приложений. Как результат, для большинства начинающих специалистов попытка 
разобраться с вопросом рискует запутать, нежели помочь.

Однако отдельный прикладной нюанс придется затронуть. Его суть в том, что при обработке HTTP-запроса контейнером 
сервлетов один и тот же фильтр может быть вызван несколько раз. В разрезе асинхронной обработки запроса это может 
произойти дважды: в момент получения запроса сервером и в момент, когда обработка запроса фактически завершена
отвечающим за это пулом потоков.

Достижима ли подобная ситуация на практике для конкретного фильтра - зависит от того, как описан его маппинг. 
Управлять им можно через сервлетный API (`web.xml`, `@WebFilter` или через настройку контекста с помощью 
`WebApplicationInitializer`). В обоих случаях можно указать, для каких значений `DispatcherType` должен срабатывать 
фильтр. Хорошая новость для классических сервлетных приложений заключается в том, что если данные настройки 
специально не трогать - фильтр будет вызываться единожды, только в ситуации, когда контейнер сервлетов получил запрос
(`DispatcherType.REQUEST`).

Однако на практике чаще всего встречаются сразу две вещи:

1. Используется асинхронная обработка запросов. В контексте `AbstractAnnotationConfigDispatcherServletInitializer` 
   этому также посвящена сноска ниже;
2. Маппинг каждого отдельного фильтра не конфигурируется, указывается лишь общий список `DispatcherType` для всех 
   фильтров. С учетом п.1 логично ожидать, что `DispatcherType.ASYNC` тоже будет присутствовать.

В целом, это не лишено смысла - ряд фильтров действительно должны вызываться несколько раз, а управлять маппингом
каждого отдельного фильтра через конфиг, находящийся отдельно от фильтра - неудобно.

> В отличие от `web.xml`, `@WebFilter` и даже `WebApplicationInitializer`, конфигурация через 
> `AbstractAnnotationConfigDispatcherServletInitializer` более автоматизирована. Во-первых, поддержка асинхронности 
> в ней по умолчанию включена (управляется через `isAsyncSupported()`). Во-вторых, если такая поддержка включена - 
> любой фильтр из `getServletFilters()` регистрируется, в том числе для вызова через `DispatcherType.ASYNC`.

Для того чтобы фактически перенести данную настройку в свою зону ответственности, Spring предоставляет абстрактный 
класс `OncePerRequestFilter`, который обычно и используется в Spring-приложениях как предок пользовательских 
фильтров. Его преимущество в том, что данный класс берет на себя обработку типичных проверок, определяющих, должен 
ли фильтр быть выполнен или нет. Чтобы разобраться с темой в полной степени, придется углубиться в семантику 
сервлетного `DispatcherType` и темы, которые под собой скрывают конкретные значения данного енама. Но на данном 
этапе предлагаю базово рассмотреть API самого `OncePerRequestFilter`.

Основная идея `OncePerRequestFilter` - гарантировать, что фильтр отработает не чаще, чем один раз на `DispatcherType`
конкретного запроса. 

> Стоит подчеркнуть, что настройки в `OncePerRequestFilter` никак не заменяют сервлетные. Они позволяют управлять 
> применением конкретного фильтра на уровне Java-кода в ситуациях, когда на сервлетном уровне фильтр уже 
> зарегистрирован как выполняющийся для нескольких `DispatcherType`. Если же фильтр и так применяется только для 
> `DispatcherType.REQUEST` (поведение по умолчанию), некоторые проверки в `OncePerRequestFilter` не имеют смысла.

`OncePerRequestFilter` представляет собой реализацию template method и предлагает пользователю реализовывать в своих
фильтрах не классический `doFilter()`, а один или несколько из предложенных `protected`-методов, позволяющих более
точечно описать логику фильтра. Нас в первую очередь будут интересовать:

- `boolean shouldNotFilter(HttpServletRequest request)`. Позволяет указать условия, при которых фильтр был пропущен. 
  Здесь может быть что угодно, но чаще всего располагают условия, не относящиеся к filter mapping напрямую. То есть 
  логика проверки по URL здесь может выглядеть странно, а вот логика проверки `Content-Type` (если фильтр делает 
  что-то только для запросов с определенным типом тела) - вполне валидный сценарий;
- `shouldNotFilterAsyncDispatch()`. Более узкий метод, валидирующий, должен ли фильтр запускаться конкретно при 
  `DispatcherType.ASYNC` (когда был подготовлен ответ на запрос обрабатываемый асинхронно). Это тот самый случай 
  второго прохода по цепочке фильтров, упомянутый выше. Значение по умолчанию - `true` (фильтр не применится);
- `doFilterInternal()`. Полезная работа фильтра. Параметры метода опущены для лаконичности, они идентичны параметрам 
  `doFilter()`;
- `shouldNotFilterErrorDispatch()`. Узкий метод валидации для `DispatcherType.ERROR`. По умолчанию - `true`;
- `doFilterNestedErrorDispatch()`. Позволяет переопределить поведение фильтра для вложенных запусков с
  `DispatcherType.ERROR`. Базовое поведение - вызов `doFilterInternal()` (обычно иное и не требуется). Это 
  метод, выделенный специально для специфики отдельных контейнеров сервлетов, которые могут запускать указанный 
  процесс в том же потоке, что и `DispatcherType.REQUEST`. Особого смысла погружаться в эту тему за пределами 
  конкретных контейнеров не имеет смысла. Параметры метода идентичны параметрам `doFilter()`;
- `isAsyncDispatch(HttpServletRequest request)`. Метод для использования в наследнике. Позволяет проверить значение 
  `DispatcherType` в текущем объекте запроса - идентично ли оно `ASYNC`;
- `isAsyncStarted(HttpServletRequest request)`. Еще один метод, использование которого предполагается в наследнике. 
  Фактически указывает, будет ли ответ закоммичен в текущем потоке. Скажем, нет смысла пытаться логировать тело 
  ответа в фильтре, обрабатывающем `DispatcherType.REQUEST`, если запрос обрабатывается асинхронно - тело банально 
  не будет записано в ответ к моменту, когда фильтр закончит свою работу в пределах текущего вызова.

Для закрепления стоит рассмотреть несколько ситуаций, в которых требуются разные значения 
`shouldNotFilterAsyncDispatch()`.

Сценарии, когда `shouldNotFilterAsyncDispatch()` должен возвращать `true`:

- Действие должно происходить строго один раз на HTTP-запрос. Счетчики, метрики количества запросов и тому подобное. 
  В чистом виде используется редко, но технически возможно;
- Различные валидации на старте обработки запроса: CORS, CSRF, IP whitelisting, валидация токенов, заголовков, 
  подписи. Сюда же относится логика различных rate limiter'ов и прочие действия, которые фактически требуют лишь 
  объекта запроса;
- Создание врапперов для запроса или ответа.

Не претендую на истину, но это покрывает бОльшую часть фильтров, которые обычно бывает необходимо реализовывать 
разработчику в том или ином виде.

Однако существуют и сценарии, когда возвращаемое значение `shouldNotFilterAsyncDispatch()` необходимо переопределить 
на `false`:

- Любая работа с контекстом, сохраняющимся на уровне потока. Обычно через `ThreadLocal`, но необязательно. Любые 
  `traceId`, Security-контексты (получение данных текущего пользователя), MDC для логгера (не затрагивали в курсе, 
  но часто применяется на практике, в том числе для трейсинга) и так далее. Очевидно, если обработка запроса 
  раскидана между несколькими потоками - необходимо в каждом потоке устанавливать необходимые значения для подобных 
  контекстов. Для какой-то специфичной логики распараллеливания или асинхронного выполнения задачи потребуется 
  дополнительная логика установки значений в контексте, но именно в случае именно с логикой асинхронной обработки
  запроса это делается через фильтры. Ниже будет приведен пример;
- Работа с ответом после его формирования. Наиболее наглядный прикладной пример - логирование ответа. В потоке, 
  обрабатывающем `DispatcherType.REQUEST` на момент завершения работы фильтра нет гарантии, что статус, заголовки 
  или тело ответа уже записаны. Зато такая гарантия есть для потока, обрабатывающего `DispatcherType.ASYNC`. Также 
  тут возможно разделение логики фильтра: один сценарий выполняется для `REQUEST`, другой - для `ASYNC`. Например, с 
  проверками через `isAsyncDispatch` или `isAsyncStarted()`.

Для закрепления рассмотрим пример фильтра, который настраивает значение ID текущего пользователя для потока. Будем 
считать, что в каком-то из предыдущих фильтров такое значение уже положили в атрибуты запроса:

```java
public class CurrentUserIdHolder {
    private static final ThreadLocal<Long> USER_ID = new ThreadLocal<>();

    public static void setId(Long id) {
        USER_ID.set(id);
    }

    public static Long getId() {
        return USER_ID.get();
    }

    public static void clear() {
        USER_ID.remove();
    }
}

public class SetUserIdFilter extends OncePerRequestFilter {
  private static final String USER_ID = "userId";

  // Первый вызов фильтра произойдет для DispatcherType.REQUEST и будет управлять значением исключительно для потока,
  // получившего запрос. Второй вызов - для DispatcherType.ASYNC, для потока, фактически обрабатывающего данный 
  // запрос и формирующего ответ
  @Override
  protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain filterChain)
          throws ServletException, IOException {
    Long userId = (Long) request.getAttribute(USER_ID);

    try {
        CurrentUserIdHolder.setId(userId);
        filterChain.doFilter(request, response);
    } finally {
        CurrentUserIdHolder.clear();
    }
  }

  @Override
  protected boolean shouldNotFilterAsyncDispatch() {
    return false;
  }
}
```

высокоуровневых фреймворков, таких, как Spring MVC, волей-неволей приходится учитывать не только сервлетную природу, 
но и особенности фреймворка. Также и сам фреймворк может попытаться решить типовые боли более низкоуровневого 
решения, как это делает `OncePerRequestFilter`.
