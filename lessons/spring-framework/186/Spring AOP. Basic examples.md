## Аспектно-ориентированное программирование

Аспектно-ориентированное программирование - парадигма программирования, нацеленная на структуризацию функциональности,
которую не удается удобно декомпозировать инструментами, существующими в других парадигмах программирования:
процедурами, функциями, методами и классами.

Если сузить определение до более понятных терминов, АОП направлена на организацию в коде сквозной функциональности. 
Этот термин мы уже использовали ранее.

Само собой, в таком ключе АОП не является самостоятельной парадигмой, ее можно применять на вспомогательную в 
дополнение к ООП или какой-либо иной императивной.

В описании бизнес-логики такой подход не имеет особого смысла - тот же ООП хорошо справляется с инкапсуляцией и 
декомпозицией в данном случае. Однако если нужна какая-то обобщенная логика для логирования, проверок прав доступа, 
управления транзакциями, кэшами или массы иных функций, которые могут потребоваться в совершенно различных частях 
программы и, зачастую, не имеют жесткой привязки к конкретной бизнес-функции или определенному слою архитектуры 
приложения - классические парадигмы часто заходят в тупик.

Разумеется, технически можно реализовать любую функциональность, не прибегая к АОП. Однако часто это будет приводить 
к запутыванию кодовой базы, в том числе - к спагетти-коду. Либо решаться через сложные или громоздкие паттерны вроде 
декораторов, все равно не избавляя окончательно от дублирования кода и неочевидных связей между различными модулями.

В разрезе Java и именно Spring AOP, аспектно-ориентированное программирование актуально в первую очередь для 
ситуаций, когда нужно либо сделать обработку аннотации (зачастую самописной), либо применять определенную логику ко 
всем классам/методам по определенному признаку. Скажем, по пакету или имени метода.

Для простых приложений такие механизмы часто будут избыточными - если кода в принципе мало, нет беды в том, чтобы 
добавить сколько-то дополнительных вызовов в нужные методы, даже несколько новых зависимостей в отдельных бинах не 
будут проблемой. Однако для крупных приложений часто требуется какое-то более обобщенное и гибко управляемое решение. 
Именно его и предлагает АОП.

Второй термин, который будет для нас актуален на данном этапе - аспект. Это, собственно, некая единица в разрезе АОП,
которая описывает логику конкретной "сквозной функции" и инструкции по применению этой сквозной функции - в каких 
случаях логика должна отработать, в принципе, когда она должна вызваться (относительно целевого метода). В случае со 
Spring AOP, аспект будет выглядеть как Java-класс, оформленный по определенным правилам.

## Подключение Spring AOP и AspectJ

В наиболее узком смысле Spring AOP доступен в любом Spring-приложении - зависимость `spring-aop` поставляется 
транзитивно для Spring Context.

Однако Spring AOP во многом опирается на более сложное и самостоятельное решение - **AspectJ**. С AspectJ в чистом 
виде мы работать не будем, лишь разберем в одной из статей некоторые ключевые моменты, связанные с ним. Однако нам 
потребуются несколько аннотаций, которые предлагает эта библиотека, также Spring в своих целях требует подключения 
одного из движков AspectJ. Зачем это нужно и как используется - разберемся позже, пока задача минимум: подключить.

Итак, нам нужны зависимости из AspectJ:

```groovy
dependencies {
    // Версия Spring не слишком критична, отношения контекста с AOP не меняются от версии к версии
    implementation 'org.springframework:spring-context:7.0.2'
    
    // Базовая библиотека для AspectJ. Нас в ней будут интересовать в первую очередь несколько аннотаций для работы 
    // с аспектами. Остальное содержимое относится к внутренней логике обработки этих аннотаций и связям с другими 
    // модулями AspectJ
    implementation 'org.aspectj:aspectjrt:1.9.24'

    // Рантайм-зависимость, без которой аспекты не сможет поддерживать сам Spring. С понятием weaving и его видами 
    // разберемся позже, пока эта зависимость нужна лишь для обработки языка выражений, который предлагает AspectJ и 
    // который Spring обязан уметь обрабатывать.
    runtimeOnly 'org.aspectj:aspectjweaver:1.9.24'
    
    // ... - Другие зависимости приложения, если есть
}
```

После подключения зависимостей потребуется сделать еще одну вещь - включить саму функциональность аспектов. Как и во 
многих иных случаях, для этого существует отдельного `@Enable`-аннотация:

```java
@EnableAspectJAutoProxy // Включаем поддержку аспектов с использованием AspectJ аннотаций
public class AppConfiguration {
}
```

На этом базовая настройка завершается и можно переходить к непосредственной практике работы с АОП. Собственно, в 99% 
Spring-проектов большей настройки и не потребуется.

## Примеры

Рассмотрим несколько популярных примеров использования Spring AOP в собственном коде.

### Логирование вызова метода

Допустим, мы хотим, чтобы перед вызовом метода, аннотированного самописной аннотацией `@LogCall` в консоль 
записывалось время вызова и имя метода. Это может выглядеть так:

```java
// Обычная маркерная аннотация
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface LogCall {
}

@Aspect // Аннотация из AspectJ - помечает класс как аспект
@Component
public class LoggingAspect {
    // Еще одна аннотация из AspectJ. Обозначает, что логика аспекта должна быть вызвана до выполнения целевого 
    // метода. Атрибут аннотации описывает условия, по которому вызов метода определяется как подходящий. В данном 
    // случае метод должен быть аннотирован @LogCall. Выражение описывается с помощью "Pointcut language" - мы с ним 
    // более подробно познакомимся позже.
    // Обычно используется полное имя аннотации (вместе с пакетом), но здесь оно опущено - допустим, аннотация и 
    // обработчик лежат в одном пакете
    @Before("@annotation(LogCall)")
    // JoinPoint - один из интерфейсов AspectJ. В общем случае является точкой входа как к непосредственному вызову 
    // целевого метода (рассмотрим в примерах ниже), так и к метаданным о целевом методе - здесь все очень схоже с 
    // reflection API. Именно в части метаданных API JoinPoint можно воспринимать как обертку над рефлексией.
    public void processAspect(JoinPoint joinPoint) {
        var now = LocalDateTime.now().format(DateTimeFormatter.ISO_DATE_TIME);
        var className = joinPoint.getSignature()
            .getDeclaringTypeName();
        var methodName = joinPoint.getSignature()
            .getName();
        // Формируем строку с указанием имени класса, имени метода и временем вызова -
        // полезная работа аспекта в данном примере
        System.out.printf("Called %s#%s() at %s\n", className, methodName, now);
    }
}

@Component
public class MyComponent {
    @LogCall
    public void test() {
       System.out.println("Test method call!");
    }
}
```

При вызове `MyComponent#test()` в логике приложения в консоль будет выведено примерно следующее:

```
Called MyComponent#test() at 2026-01-01T00:00:00.000000
Test method call!
```

Метод, описывающий поведение аспекта (в нашем случае - `LoggingAspect#processAspect()`) могут называть **advice**. 
В русском комьюнити термин обычно не переводят, а пользуются транслитерацией - эдвайс. Вероятно, "совет" звучит 
недостаточно круто:) 

### Вычисление времени выполнения метода

Чуть более сложный пример, также более наглядный в части работы с `JoinPoint` - вычисление времени, затраченного на 
обработку отдельного метода. Может выглядеть так:

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ExecutionTimer {
}

@Aspect
@Component
public class ExecutionTimerAspect {
   // @Around - аннотация AspectJ, означающая выполнение аспект "вокруг" целевого метода. Логичное следствие - 
   // ответственность за вызов целевого метода лежит на разработчике аспекта.
   // Как и в прошлом примере, аспект срабатывает для методов, помеченных определенной аннотацией
   @Around("@annotation(ExecutionTimer)")
   // ProceedingJoinPoint - наследник JoinPoint. Ключевая особенность - позволяет сделать "вызов" целевого метода. 
   // Мы не знаем заранее, есть ли у целевого метода throws-блок, поэтому обязаны обрабатывать Throwable. В 
   // данном случае - просто пробрасываем его дальше по стеку вызовов
   public Object processAspect(ProceedingJoinPoint joinPoint) throws Throwable {
      var className = joinPoint.getSignature()
              .getDeclaringTypeName();
      var methodName = joinPoint.getSignature()
              .getName();

      // Полезная нагрузка аспекта описана максимально наивно. Замеряем время до и после вызова, считаем разность, 
      // записываем ее в консоль
      var start = System.nanoTime();
      
      // У целевого метода могут быть параметры и возвращаемые значения. В данном случае мы не указываем параметры 
      // для proceed() (если они есть у метода - будут проброшены без изменений), но у нас есть техническая возможность 
      // обработать их по своему усмотрению, в том числе заменить
      // Аналогично и с возвращаемым значением. Мы не подменяем его в этом примере, но технически можем. Для 
      // void-методов в result будет записан null
      var result = joinPoint.proceed();

      var finish = System.nanoTime();
      var duration = finish - start;

      System.out.printf("Call of %s#%s() takes %d nanos\n", className, methodName, duration);

      return result;
   }
}

@Component
public class MyComponent {
   @ExecutionTimer
   public void test() {
      System.out.println("Test method call!");
   }
}
```

Результат вызова `MyComponent#test()`:

```
Test method call!
Call of MyComponent#test() takes 208000 nanos
```

На `ProceedingJoinPoint#proceed()` стоит остановиться отдельно. В утрированной формулировке этот метод просто 
вызывает целевой, вокруг которого и был запущен аспект. Но семантически он ближе к `HttpFilter#doFilter()` в Servlet 
API. То есть фактически данный метод осуществляет передачу управления - будет вызван либо следующий аспект в 
цепочке, либо, если все аспекты пройдены - целевой метод.

### Проверка прав доступа

Еще один достаточно распространенный сценарий - проверка прав пользователя на выполнение определенного метода. 
Такого рода логика существует и из коробки в Spring Security, и, в некоторых случаях, пишется самостоятельно, если 
базовое решение не подходит по тем или иным причинам.

Пример ниже местами условный, но суть передает верно:

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface HasRole {
    // Сюда будем передавать информацию о роли, которая требуется для получения доступа к аннотированному методу
    String value();
}

@Aspect
@Component
public class HasRoleAspect {
   @Around("@annotation(HasRole)")
   // Мы можем передавать объект аннотации параметром в метод. В целом, сигнатура метода, обрабатывающего аспект - 
   // широкая тема, которую мы детально рассмотрим позже 
   public Object processAspect(ProceedingJoinPoint joinPoint, HasRole hasRoleData) throws Throwable {
      var className = joinPoint.getSignature()
              .getDeclaringTypeName();
      var methodName = joinPoint.getSignature()
              .getName();

      var expectedRole = hasRoleData.value();
      var actualRole = SecurityContext.getCurrentRole();

      // Если if сработает - целевой метод вообще не будет вызван. Что нам и нужно в данном случае
      if (!expectedRole.equals(actualRole)) {
         throw new RuntimeException("User has no access to %s#%s()".formatted(className, methodName));
      }

      return joinPoint.proceed();
   }
}

public class SecurityContext {
   public static String getCurrentRole() {
      // ... - Какая-то реализация, возвращающая роль текущего пользователя. Как она будет работать - неважно в 
      // контексте текущей статьи 
   }
}

@Component
public class MyComponent {
   @HasRole("Admin")
   public void test() {
      System.out.println("Test method call!");
   }
}
```

### Привязка аспекта без аннотации

Важно понимать, что Spring AOP и, в частности, AspectJ expression - достаточно мощные и гибкие инструменты. Все 
примеры выше ориентировались на аннотацию и позволяли привязываться именно к ней. Но жизнь намного более 
многообразна, в различных условиях может потребоваться вызывать аспект для каждого метода, относящегося к 
определенному классу, или каждый метод, имеющий параметр определенного типа или... Множество различных, в том числе 
комбинируемых условий.

Ниже мы рассмотрим аспект, который будет вызываться для всех методов, расположенных в пакете `com.walking`:

```java
@Aspect
@Component
public class LoggingAspect {

    @Before("execution(* com.walking.*.*(..))")
    public void processAspect(JoinPoint joinPoint) {
        var now = LocalDateTime.now().format(DateTimeFormatter.ISO_DATE_TIME);
        var className = joinPoint.getSignature()
            .getDeclaringTypeName();
        var methodName = joinPoint.getSignature()
            .getName();

        System.out.printf("Called %s#%s() at %s\n", className, methodName, now);
    }
}

// Будем считать, что данный класс находится в пакете com.walking
@Component
public class MyComponent {
   public void test() {
      System.out.println("Test method call!");
   }
}
```

При вызове `MyComponent#test()` до вызова целевого метода отработает логика эдвайса. Несмотря на то, что никакой 
аннотации или иного явного маркера в самом классе нет.

Выражение `execution(* com.walking.*.*(..))` вероятно вызовет много вопросов на данном этапе. Более подробно с 
форматом таких выражений будем знакомиться в отдельной статье - едва ли мы покроем экспертный уровень, но хотя бы 
подобные типовые конструкции перестанут вызывать ступор. Пока же ограничимся кратким описанием:

- `execution()` - собственно, указывает на способ поиска целевых методов. В предыдущих примерах это был `annotation()`
  с очевидным поведением привязки по аннотации, здесь же привязка работает по факту "вызова" (читай, существования) 
  метода;
- Первый `*` описывает возвращаемый тип метода. В данном случае нас устраивает любой тип возвращаемого значения. 
  Очевидно, при желании этот параметр можно задать более конкретно;
- `com.walking`. Обозначение пакета. Здесь он задан жестко, вложенные пакеты не учитываются;
- Следующая `*` (после `com.walking.`). Имя класса. В данном случае - любой класс без дополнительного фильтра по имени;
- Последняя `*`. Имя метода. Здесь - любое;
- `(..)` - секция, отвечающая за описание параметров целевого метода. Такая форма записи соответствует любым 
  параметрам, включая их отсутствие.

На этом мы завершаем разбор примеров. Мы рассмотрели лишь самые простые способы работы со Spring AOP. Чтобы 
расширять знания дальше, необходимо расширить теоретический базис по этому разделу, чем и займемся в следующих статьях. 

## Базовые ограничения

Spring AOP - вероятно, один из самых дружелюбных и легких для старта инструментов как работы с АОП, так и в более 
узком смысле - работы с собственными аннотациями. Многое из того, что в иных условиях пришлось бы делать вручную 
(вроде сканирования classpath в поиске мест использования аннотации) сделано за нас, с предоставлением удобного и 
лаконичного API.

Однако за все приходится платить. В данном случае платой выступает механизм обработки аспектов в Spring AOP. Аспекты 
применяются через прокси-классы, создаваемые для каждого отдельно взятого бина. Детали такого проксирования 
разбирались в прошлой статье. В разрезе текущей темы это дает несколько прикладных ограничений именно для аспектов:

1. Аспекты могут применяться только для Spring-бинов. Для AspectJ в базовом представлении это не так, но в Spring 
   AOP - иного не дано;
2. Аспекты могут работать только с неприватными и не финальными методами. С приватными и иными не публичными есть 
   очевидное ограничение при динамическом проксировании (приватные методы интерфейсов появились относительно поздно), с 
   финальными - не менее очевидное ограничение со стороны CGLIB. Поэтому в базовом сценарии метод должен быть 
   обозначен как `public`* и НЕ помечен `final`, чтобы к нему могли применяться аспекты;
3. Статические методы тоже не могут быть обработаны аспектами. Идея не нова - нет механизма переопределения, нет и 
   адекватного способа создать прокси-класс. Отсюда нет и возможности привязать аспект. По крайней мере в пределах 
   Spring AOP;
4. Неоднократно упомянутое ранее ограничение self-reference. Если мы вызовем из класса метод того же класса, к 
   которому должен примениться аспект - эдвайс не сработает, ведь мы обратимся к объекту target-класса,
   не обращаясь к прокси, где и находится логику обработки аспекта. Решения проблемы то же - self-inject. О 
   некоторых альтернативных способах обойти данную проблему поговорим позже, имея более глубокие знания в Spring AOP.

> *Технически, `protected` и `package-default` методы тоже могут быть обработаны через Spring AOP, если 
> проксирование реализовано через CGLIB. Для JDK Dynamic Proxy это не будет работать по очевидным причинам. 
> Соответственно, если по какой-то причине необходимо применение аспектов для непубличных методов - стоит убедиться, 
> что проксирование работает именно через CGLIB. Например, через установку
