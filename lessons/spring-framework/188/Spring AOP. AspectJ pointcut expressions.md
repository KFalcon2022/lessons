# Spring AOP. AspectJ pointcut-выражения

Данная статья завершает подраздел Spring AOP в текущем курсе. Ее цель - дать системное представление о языке
pointcut-выражений, предоставляемом AspectJ. При этом ширина и глубина изучения темы ограничена с нескольких сторон:

- Мы рассматриваем данный язык через призму Spring AOP, где он может использоваться довольно ограничено. Как минимум,
  нам недоступны возможности языка за пределами перехвата методов, есть ограничения и по зонам видимости
  перехватываемых методов. В целом, мы будем рассматривать лишь те возможности, которые доступны в Spring AOP при
  runtime weaving;
- У нас не стоит задачи ни изучить компилятор для pointcut-выражений, ни написать собственный. Соответственно, нет
  необходимости ни рассматривать, как эти выражения обрабатывает сам Spring AOP (и, далее, AspectJ-парсер)
  внутри, ни опускаться в работу синтаксического дерева.

В общем случае, большинство Java-разработчиков используют довольно простые pointcut'ы, в которых AspectJ-выражения
используются интуитивно и, преимущественно, по образцу. Из-за этого данную статью можно считать дополнительной и
даже необязательной для изучения - большая часть практического применения Spring AOP в коммерческой разработке
вполне достижима по итогам изучения двух предыдущих статей. Однако интуитивный подход часто дает сбои при
необходимости реализовать что-то минимально нестандартное - здесь на помощь приходит понимание основ синтаксиса и
грамматики языка pointcut-выражений.

В конечном итоге, нам необходимо разобраться с основными операторами и зоной их применения в языке. В этом помогут, в
том числе, изученные ранее примеры.

> За основу взята документация Spring
> ([ссылка](https://docs.spring.io/spring-framework/docs/4.2.x/spring-framework-reference/html/aop.html))
> и некоторые иные источники, включая гайды AspectJ
> ([ссылка](https://eclipse.dev/aspectj/doc/released/progguide/index.html)). При желании углубиться в тему,
> рекомендую начать с этих ресурсов. Помните, что Spring AOP поддерживает лишь часть возможностей AspectJ.

## Pointcut Designators

**Designator** - оператор языка выражений AspectJ, по своей логике напоминающий функцию (метод в терминах Java). Он 
определяет семантику, в соответствии с которой будут обрабатываться переданные в него параметры. В примерах мы 
встречали некоторые из designator'ов: `execution`, `@annotation`, `args`. Ниже подробнее разберем те из 
designator'ов, которые поддерживаются в Spring AOP.

### `execution()` 

Вероятно, наиболее общий из всех designator'ов, поддерживаемых Spring AOP. Позволяет привязаться к join point'у по 
паттерну, описывающему конкретный метод - от модификатора доступа, типа возвращаемого значения и принадлежности к 
определенному пакету или классу до имени метода и его параметров. Выглядит примерно так:

```
execution(* com.walking..*.*(..))
```

Позже мы разберем значения отдельных операторов более подробно - `*`, `..`, `+` (отсутствует в примере выше). 
Непосредственно данное выражение можно трактовать как "любой метод, расположенный в пакете `com.walking` или его 
подпакетах". При этом стоит помнить, что Spring AOP может обрабатывать только `public` и, в случае с CGLIB-proxy, 
`protected` и package-private методы, если не включен compile-time или load-time weaving.

Похожие конструкции будут встречаться и в некоторых иных designator'ах, поэтому разберем чуть подробнее, какая часть 
выражения внутри `execution()`. Однако выберем еще более показательный пример:

```
public * com.walking..*.*(..)
```

В отличие от предыдущего, здесь выражение ограничено только публичными методами и оно позволяет описать почти все 
возможности данного паттерна:

1. Модификатор доступа методов, которые соответсвуют pointcut. В примере выше - только `public`-методы. Данный 
   сегмент выражения может быть опущен, что будет соответствовать "методы с любыми модификаторами";
2. `*`. Отвечает за тип возвращаемого значения описываемого метода. В данном случае - любые типы, включая `void`. 
   Если необходим конкретный тип - его можно указать явно. Например: `public String com.walking..*.*(..)` - 
   публичные методы, возвращающие строку;
3. `com.walking..`. Пакет `com.walking` и его подпакеты. `..` в данном случае означает любой путь (или участок пути).
   То есть вообще любые методы любых классов можно описать как `* ..*.*(..)`*. В свою очередь, выражение
   `public * com.walking.*.*(..)` означало бы жесткую привязку к пакету `com.walking` НЕ включая подпакеты;
4. `*` (после `..`). Описывает шаблон имени класса. В данном случае - любое имя класса. Здесь возможно указать и 
   конкретное имя (`com.walking..MyClass.*(..)`), так и некий промежуточный формат - шаблон имени, например
   `com.walking..*.My*.*(..)` (все классы, имя которых начинается с `My`) или `com.walking..*Class.*(..)` (все классы, 
   имя которых заканчивается на `Class`);
5. `*` (перед `(..)`). Шаблон имени метода. Логика такая же, как и для имен классов - `*` обозначает любое имя, 
   можно также использовать конкретный литерал или шаблон с частичным совпадением имени;
6. `(..)`. Шаблон описания параметров. Описывается внутри скобок, как и параметры при описании метода. `..` в данном 
   случае означает "любые параметры или их отсутствие". Если параметров быть не должно вообще - используют пустые 
   скобки (`com.walking..*.*()`). Также можно указать один или несколько параметров для строгого совпадения по типу 
   (`com.walking..*.*(String)` - один строковый параметр, `com.walking..*.*(int, int)` - два параметра типа `int`). 
   Наконец, доступны комбинации, когда проверяется совпадение одного нескольких параметров, а остальные не имеют 
   значения (`com.walking..*.*(String, ..)` - первый параметр строковый, остальные не имеют значения). Наконец, 
   вместо типа может использоваться `*`, что значит, что параметр должен присутствовать, но его тип неважен:
   `com.walking..*.*(String, *)` - первый параметр типа `String`, второй - любого типа.

> *Выражение `* ..*.*(..)`можно описать проще. Сегмент `..*.*` фактически означает "любой пакет(`..`), любое имя 
> класса(`*`), любое имя метода (вторая `*`)". Более простой формой записи будет `* *(..)`. Подробное описание 
> "любого типа и имени метода" схлопывается до одной `*`.

Важный момент, касающийся работы с параметрами - `execution()` сопоставляет сигнатуру метода, а не фактические типы 
во время выполнения. То есть те параметры метода, совпадения по которым он ожидает, должны быть заданы в сигнатуре.
Скажем, если выражение выглядит как `com.walking.*.*(Object)`, то `com.walking.MyClass#doSth(Object o)` подойдет, а
`com.walking.MyClass#doSth(String o)` - уже нет, даже несмотря на то, что `String` - наследник `Object`.

Несложно заметить, что AspectJ-выражение можно описать весьма гибко, для большинства сценариев использования - 
избыточно гибко. Очевидно, что чем сложнее выражение, тем более специализированный аспект его использует. Поскольку 
аспекты обычно используются для максимально общей сквозной функциональности - большая часть рассмотренных выше 
возможностей банально не востребована в задачах среднего разработчика.

### `args()` 

Designator `args` может использоваться в двух целях:

1. Привязка параметров перехватываемого метода к параметрам эдвайса. Это рассматривалось в предыдущей статье и имеет 
   смысл в первую очередь для удобства обработки;
2. Ограничение pointcut. Также, как в `execution()` можно ограничивать подходящие методы по типам параметров, так и 
   здесь, с небольшими отличиями.

Первый пункт уже был разобран ранее, осталось рассмотреть второй. `execution()` - комплексный designator, который 
иногда выглядит слишком громоздко. В таком случае может иметь смысл описание шаблона параметров pointcut в `args`. 
Синтаксис идентичен тому, который описывается внутри параметров метода в `execution()`:

```
execution(* com.walking..*.*()) -> args() 
execution(* com.walking..*.*(String)) -> args(String) 
execution(* com.walking..*.*(int, int)) -> args(int, int) 
execution(* com.walking..*.*(String, ..)) -> args(String, ..) 
execution(* com.walking..*.*(String, *)) -> args(String, *)
```

Однако в обработке `execution()` и `args()` есть корневое отличие, из-за которого нельзя рассматривать `args()` лишь 
как более узкий designator относительно `execution()`. `execution()` работает со статикой - он смотрит на сигнатуру 
метода в коде. `args()`, в свою очередь, обрабатывается во время исполнения. Отсюда возникают ситуации, при которых 
эти designator'ы будут обладать различным поведением.

Разберем на примере:

```java
public class MyClass {
    public void method1(String s) {}

    public void method2(Object o) {}
}
```

Один из методов принимает на вход строку, второй - любой объект. Рассмотрим сценарий, в котором один pointcut 
описан как `execution(* *(String))` - любой метод со строковым параметром, а второй - как `args(String)` - в целом, 
тоже любой метод со строковым параметром.

Если мы сделаем два вызова:

```java
myClass.method1("example");
myClass.method2("example"); // Строка - наследник Object, можно использовать в этом методе
```

То поведение будет различным:

- Pointcut с `execution()` будет использоваться только для `method1()`, потому что строковый параметр является 
  частью сигнатуры метода;
- Pointcut с `args()` будет считать подходящими оба вызова - в обоих случаях _фактически_ был использован строковый 
  параметр.

В ситуациях, когда pointcut описывается для методов бизнес-логики без полиморфных параметров разницы обычно нет. 
Если pointcut более общий или параметры метода заданы интерфейсами (или типами, имеющими наследников) - поведение 
может отличаться. И здесь, в зависимости от семантики связанных эдвайсов необходимо корректно выбрать, что именно 
нас интересует - сигнатура или фактические значения при вызове.
 
### `within()` 

Еще один designator, который можно рассматривать как подвид `execution()`. Если `execution()` описывает шаблон 
метода, то `within()` - исключительно шаблон типа, все методы* которого будут попадать в pointcut.

> *С поправкой на особенности выбранного типа weaving и типа используемого прокси.

То есть `within()` подходит для ситуаций, когда `execution()` допускает все пункты сразу:

- Любые модификаторы доступа метода;
- Любые типы возвращаемого значения метода:
- Любое имя метода;
- Любые параметры метода;
- Нет подводных камней в виде наследования или других изменений, влияющих на семантику - об этом подробнее 
  рассказано ниже.

Может показаться, что допущений слишком много, однако на практике это один из популярных сценариев. Чтобы не писать 
пачку `*` и `..`, можно их просто проигнорировать:

```
execution(* com.walking..*.*(..))           == within(com.walking..*)
execution(* com.walking.*.*(..))            == within(com.walking.*)
execution(* com.walking..MyClass.*(..))     == within(com.walking..MyClass)
execution(* com.walking..*.My*.*(..))       == within(com.walking..*.My*)
execution(* com.walking..*Class.*(..))      == within(com.walking..*Class)
```

Как и `execution()`, `within()` анализирует статитику. Но как и в случае с `args()`, нельзя рассматривать один 
designator как частный случай другого.

`within()` смотрит, где декларируется метод, а не его фактическую принадлежность к одному и тому же типу. Пока 
рассматриваемые классы и их методы не выстраиваются в иерархии - никакой разницы нет. Однако в случае с 
наследованием могут быть нюансы:

```java
package com.walking.declare

@Component
public class ParentClass {
    public void doSth() {}
}
// -------------------
package com.walking.impl

@Component
public class ChildClass extends ParentClass {
}
```

Теперь если мы определим поинткаты `execution(* com.walking.declare.ParentClass.*(..)` и
`within(com.walking.declare.ParentClass)` - мы заметим, что для вызова `childClass.doSth()` будет срабатывать лишь 
`execution()`. Это вызвано тем, что `within()` оперирует лишь декларациями, в то время как `execution()` оперирует 
фактическими вызовами методов, независимо от того, делает их объект класса, который эти методы декларирует, или 
объект класса-наследника. Мы уже сталкивались с похожей двойственностью при знакомстве с Reflection API. 

В зависимости от логики конкретного эдвайса может потребоваться однозначный выбор - или `execution()`, или 
`within()`. В отдельных случаях потребуется дополнительный оператор `+` или же использование другого designator'а - 
`target()` - оба инструмента рассмотрим ниже.

### `this()` и `target()`

Здесь разберем сразу два designator'а, поскольку попытки рассматривать их по отдельности чаще всего приводят к путанице.

В наивном описании, и `this()`, и `target()` являются относительно непопулярными альтернативами `within()`, 
анализирующимися во время выполнения. Как и `within()`, они ограничивают зону применения поинтката по типу. Однако если
`within()` оперирует Java-типом и анализирует в первую очередь код - `this()`, и `target()` оперируют терминами
обрабатываемого типа и типом target object. Синтаксис у всех трех идентичен:

```
within(com.walking..MyClass)
this(com.walking..MyClass)
target(com.walking..MyClass)
```

Объяснение может показаться путаным, но лишь на первый взгляд.

`within()` проверяет класс, внутри которого объявлен метод. Все методы, декларированные в предках, игнорируются. 
`target()` оперирует целевым объектом и его принадлежностью к типу. Все доступные методы объекта будут попадать под 
условие pointcut независимо от того, какими классами они декларируются. Наконец, `this()` всегда оперирует 
прокси-типом, что ставит семантику designator'а в зависимость от используемого типа прокси.

Ключевое, что необходимо вынести из текущего пункта в разрезе Spring AOP:

- Если необходимо обрабатывать все методы бина, а не конкретного типа - проще использовать `target()`;
- Если нужен статический анализ и обрабатывать методы, декларируемые в типе - `within()` или `execution()` (разница 
  описана в предыдущем пункте);
- `this()` используется в первую очередь для инфраструктурных аспектов, связанных непосредственно с фреймворком и 
  имеет зависимость от типа прокси - на практике не нужен почти никогда. Специфика его работы разобрана подробнее ниже. 

Рассмотрим пример небольшой иерархии:

```java
package com.walking;

public interface MyService {
    void method();
}

@Service
public class MyServiceImpl implements MyService {
    @Override
    public void method() {}
}
```

И нескольких поинткатов:

```
target(com.walking.MyServiceImpl)
target(com.walking.MyService)
this(com.walking.MyService)
this(com.walking.MyServiceImpl)
```

Какие из них сработают при вызове `myService.method()` при условии, что бин был корректно внедрен и Spring AOP в 
принципе видит этот вызов?

Здесь и начинается самое интересное. По сути, во всех четырех случаях АОП-движок будет перехватывать вызов метода и 
проверять его соответствие указанному в выражении типу через `instanceof`. Но результаты будут отличаться:

- `target(com.walking.MyServiceImpl)` и `target(com.walking.MyService)`. Здесь совпадение будет всегда. Целевой 
  объект - экземпляр класса `MyServiceImpl`. А значит является и экземпляром `MyService`;
- `this(com.walking.MyService)`. Тут, в принципе, тоже всегда будет происходить совпадение. Бин будет либо 
  проксироваться через CGLIB, а значит тип прокси будет наследником `MyServiceImpl`, а значит, по цепочке, и 
  наследником `MyService`. Либо проксирование будет происходить через JDK dynamic proxy и фактический тип будет 
  что-то вроде `Proxy123`, напрямую реализующий `MyService`;
- `this(com.walking.MyServiceImpl)`. Здесь самое интересное, но работает парадигма, описанная выше. Для прокси через 
  CGLIB совпадение будет, ведь конечный тип - наследник `MyServiceImpl`. А вот для JDK dynamic proxy такой поинткат 
  не будет выдавать совпадения - ведь прокси наследуется напрямую от `MyService`, класс `MyServiceImpl` находится 
  вне этой иерархии и, соответственно, поинткат не сработает.

Таким образом, при работе с `this()` всегда необходимо четко понимать, какой тип проксирования используется для 
конкретного бина. Однако на практике существует не так много сценариев, когда необходимо работать строго с 
прокси-типом, а не с целевым объектом. Что автоматически отдает приоритет использования более предсказуемому 
`target`. `this()` остается для узких сценариев, в котором либо существует несколько прокси-оберток и необходимо 
выполнить логику эдвайса строго в определенном порядке, либо что-то еще заставляет жестко привязывать логику эдвайса 
именно к прокси-объекту.

### `@annotation()` 

Этот и последующие designator'ы относятся к обработке аннотаций и легче воспринимаются новичками. При этом они куда 
популярнее на практике, по крайней мере, для типовых задач.

Обратите внимание, что некоторые designator'ы имеют одинаковые буквенные обозначения, но отличаются 
наличием (или отсутствием) префикса `@`.

`@annotation()` указывает, что интересующие поинткат методы должны быть помечены аннотацией, указанной в скобках:

```
@annotation(MyAnnotation)
```

При этом требуется указывать полное (вместе с пакетом) имя аннотации, за исключением случаев, когда аннотация и 
аспект находятся в одном пакете.

Этот designator неоднократно упоминался в предыдущих статьях, поэтому добавить здесь особо нечего. Однако можно 
отметить общую для всех `@`-designator'ов особенность. Как `args()` позволяет привязать параметр целевого метода к 
параметру эдвайса, любой `@`-designator позволяет привязать к параметру эдвайса объект упоминаемой им аннотации. Это 
было продемонстрировано в предыдущих статьях для `@annotation()`, аналогично может быть использовано для любого 
designator'а ниже.

### `@within()` 

Проверяет аннотацию над типом (классом или интерфейсом). Как и `within()`, анализируется статически.

```java
@Component
@MyAnnotation
public class MyComponent1 {
    public void method() {}
}

@Component
@MyAnnotation
public class MyComponent2 {
    public void method() {}
}
```

Для поинтката `@within(MyAnnotation)` будет происходить совпадение на каждый вызов любого метода классов 
`MyComponent1` и `MyComponent2`.

### `@target()` 

Также проверяет аннотацию над типом. Но исключительно над типом целевого объекта. Таким образом, если аннотация 
объявлена в интерфейсе (или суперклассе) - проигнорирует ее, поскольку целевой объект всегда относится к классу. 
Поведение `@within()` в аналогичной ситуации будет зависеть от способа внедрения - при внедрении через тип 
интерфейса (или суперкласса) аннотация над интерфейсом (или суперклассом) будет учитываться, при внедрении по 
целевому типу - нет. Как и в случае с `within()`, здесь проявляет себя концепция деклараций.

```java
@MyAnnotation
public interface MyComponent {
    void method();
}

@Component
public class MyComponentImpl implements MyComponent {
    @Override
    public void method() {}
}
```

В данном случае `@target(MyAnnotation)` не будет срабатывать для вызовов `MyComponentImpl#method()` независимо от 
типа проксирования и способа внедрения бина. Исключение - если `MyAnnotation` обозначена как `@Inherited`-аннотация.

### `@args()` 

Довольно специфический designator, проверяющий аннотации, примененные к актуальным типам параметров. Сценарий 
адекватного практического применения придумать сложно, поэтому обойдемся без него - не уверен, что сам хотя бы раз 
видел использование этого designator'а за пределами учебных примеров.

Идеологически выглядит так:

```java
@MyAnnotation
public class Model {
}

@Component
public class MyComponent {
    public void method1(Model model) {}
    
    public void method2(Object object) {}
}
```

Pointcut-выражение:

```
@args(@MyAnnotation)
```

`@args()` анализируется во время исполнения. Поэтому при вызове `MyComponent#method1()` поинткат будет валидным всегда,
за исключением ситуаций, когда фактический тип аргумента является наследником `Model`, не помеченным соответствующей 
аннотацией. Для `MyComponent#method1()` - только в ситуациях, когда параметром передан объект `Model` или иного 
класса, помеченного `@MyAnnotation`.

На этом мы завершаем знакомство с designator'ами и можем идти дальше.

> За пределами рассмотренных designator'ов остается перечень тех, которые не попали в статью, так как не поддерживаются
> в Spring AOP, при этом существуют в AspectJ. Их реализация невозможна на базе proxy-based АОП: `call()`,
> `initialization()`, `get()`, `set()` и другие.

## Паттерны и wildcard

Как мы уже выяснили выше, кроме designator'ов стоит уметь работать и с несколькими иными операторами. Данный пункт 
описывает те из них, которые используются в шаблонах для designator'ов вроде `execution()`, `within()` и им подобных.
Отчасти дублирует описанное выше, но более лаконично - на случай, если потребуется в справочном формате.

### `*`

В общем случае обозначает "любой токен". В остальном зависит от контекста применения:

- Для типа возвращаемого значения (в `execution()`) означает любой тип;
- В шаблоне типа данных (описании пакета или класса) означает любой литерал, в том числе можем использоваться как часть
  шаблона: как `com.walking.*.*(..)`, так и `com.walk*.*.*(..)` (попадает и `com.walking`, и `com.walker` 
  и другие вариации имени пакета, начинающегося с `walk`), и `com.walking.*Service.*(..)`;
- Для шаблона имени метода - аналогично логике предыдущего пункта;
- Для параметров метода означает один параметр любого типа, может комбинироваться с другими синтаксическими 
  возможностями: `com.walking.*.*(*)` - строго один параметр любого тип, `com.walking.*.*(String, *)` - первый 
  параметр должен быть строкой, второй должен существовать, тип не важен.

### `..`

Имеет два значения в зависимости от места применения, оба описаны в разборе `execution()`.

Первый вариант - обозначение вложенных пакетов в шаблоне типа: `com.walking..`. Обозначает, что подходящий тип может 
быть как в указанном пакете (`com.walking`), так и в его подпакетах любой глубины вложенности.

Второй вариант означает наиболее общее ограничение параметров метода - любые параметры или их отсутствие:
`* com.walking.MyClass.*(..)`. Обычно используется в ситуациях, когда состав параметров не имеет значения для 
поинтката. В более сложных ситуациях может быть частью более комплексного выражения. Скажем, здесь:
`* com.walking.MyClass.*(int, .., String)` ожидается, что первым параметром метода будет `int`, последним - `String`,
а между ними будет любое (включая ноль) число параметров, типы которых мы не валидируем в данном поинткате. 

### `+`

Применяется для шаблона имени класса и обозначает "подтипы".

То есть, если выражение `within(com.walking.MyClass)` означает принадлежность классу `com.walking.MyClass`, то 
выражение `within(com.walking.MyClass+)` будет включать как `com.walking.MyClass`, так и всех его наследников,
независимо от того, в каких пакетах эти наследники расположены.

Соответственно, могут существовать и более обобщенные шаблоны, скажем, `com.walking.My*+` или даже `com.walking.*+`.
Актуально для шаблонов, так или иначе оперирующих именем класса, метод которого перехватывается - `execution()`, 
`within()` и т.д.

## Логически операции и комбинация условий

Как и в других языках выражений (например, SpEL), AspectJ expression language поддерживает логические операции. Это 
может быть актуально, когда аспект должен примениться адресно по совпадению нескольких условий или же НЕ совпадению 
с каким-то условием.

Здесь все достаточно просто, сами логические операторы практически идентичны тем, которые используются в Java

- `&&`. Логическое И. Требует выполнения всех условий, объединенных таким оператором. Ранее мы встречали несколько 
  примеров, например - `@annotation(MyAnnotation) && args(myArg)`;
- `||`. Логическое ИЛИ. Хотя бы одно из условий должно быть выполнено, чтобы pointcut считался соответствующим. 
  Например, использование одной из аннотаций: `@annotation(MyAnnotation) || @annotation(MyAlternativeAnnotation)`;
- `!`. Отрицание (НЕ). Не совпадение заданного условия. Скажем, все классы, не принадлежащие заданному пакету (и его 
  подпакетам):
  `!within(com.walking.example.ignoring..*)`. 

Ключевое отличие от Java - нет сепарации на обязательные и опциональные проверки (`|` и `||`, `&` и `&&`). В этом 
просто нет необходимости, т.к. сами условия выражения не несут побочных эффектов, в отличие от логических выражений 
в Java, где выполнение операции может быть необходимо не только для получения булевого значения, но и для каких-либо 
действий, выполняемых при вычислении результата выражения. Также отсутствует XOR (исключающее ИЛИ), которое в Java 
доступно через оператор `^`.

Возможна также комбинация трех и более условий, в том числе с несколькими логическими группами. Логика определения 
таких групп и их приоритетов аналогична таковой в Java-выражениях:

```
(@annotation(MyAnnotation) || @annotation(MyAlternativeAnnotation)) && !within(com.walking.example.ignoring..*)
```

В данном случае poincut'у будут соответствовать все методы, помеченные аннотациями `@MyAnnotation` или 
`@MyAlternativeAnnotation`, кроме методов классов, расположенных в пакете `com.walking.example.ignoring` (и его 
подпакетах).

## Именованные pointcut

Эта функциональность фактически уже была разобрана в предыдущей статье. Использование аннотации `@Pointcut` 
позволяет создавать собственные именованные поинткаты, чье использование синтаксически будет напоминать отдельный 
designator.

Здесь остается лишь отметить, что это не какая-то отдельная возможность Spring AOP, а часть языка выражений AspectJ.

## Группировка и синтаксические операторы

По сути, разбор выше является достаточным. Для чистоты картины остается обозначить, что за пределами статьи остаются
операторы, чье назначение кажется очевидным - они могли фигурировать в примерах, но отдельно не описаны выше:

- скобки (`()`) для обозначения принадлежности параметров определенному designator'у или группировки сложных условий
  (при использовании логических операторов). Тут прямая аналогия с Java и большинством C-подобных языков
  программирования, где скобки используются и для описания параметров функции, и для определения групп условий;
- Запятая (`,`) для разделения параметров при работе с аргументами;
- Одна точка (`.`) для разделения сегментов пакета, отделения шаблона пакета от шаблона имени класса, также для 
  отделения шаблона имени метода.

В этих случаях смысл операторов идентичен или крайне похож на их смысл в Java, поэтому какого-то особого внимания им 
не уделено.
