# Spring MVC. Основные аннотации и практика использования

Основная цель данной статьи - поверхностное знакомство с аннотациями, которые в большей или меньшей степени будут
полезны при работе со Spring MVC. В общем-то, эти же аннотации используются и в реактивной альтернативе - Spring
WebFlux, то есть можно говорить о том, что почти любая обработка HTTP-запросов в Spring-приложениях будет связана с
рассматриваемыми сегодня аннотациями.  

Дополнительно упомянуты несколько аннотаций из того, что обычно понимают под Spring MVC в узком смысле - работа с 
формами, JSP или шаблонизаторами. Возможно, мы рассмотрим эту тему чуть подробнее в дополнительной статье - как 
исторический срез, редко актуальный в современных проектах. Пока лишь рассмотрим отдельные аннотации, проведя 
параллели с Servlet API.

Теперь о том, чего в статье не будет:

- Каких-то супер-узких аннотаций или сценариев использования. В целом, список ниже формировался по принципу "я это 
  хотя бы раз использовал и это кажется полезным". Некоторые аннотации имеют альтернативы, какие-то могут означать 
  разные несколько вещи в зависимости от формата использования или неочевидные ограничения. Пытаться описать их все за 
  раз - бесперспективно;
- Глубокого разбора, как обрабатывается та или иная аннотации. Отчасти это будет в следующих статьях, но не здесь.

Таким образом стараемся сохранить практико-ориентированный формат - сначала смотрим, что предлагает фреймворк, затем 
разбираемся, как он это реализует - на уровне концепций или в деталях, с погружением в кодовую базу.  

## Маппинг Запроса

### Регистрация контроллеров

Как мы знаем из предыдущей статьи, Spring MVC работает не с сервлетными классами, а вводит более высокий уровень 
абстракции - контроллеры.

Обозначить класс как контроллер можно двумя способами - использовать либо аннотацию `@Controller`, либо ее более 
продвинутого наследника - `@RestController`. В обоих случаях эти аннотации будут регистрировать класс в качестве 
компонента и дополнительно (в логике Spring MVC) - как слушателей HTTP-запросов - то, что и называется термином 
"контроллер" в данном случае.

> Технически, `@Controller` - это лишь маркер в Spring Core, который может быть применен к любому классу. 
> Дополнительную логику ему дает уже Spring MVC. Он же (точнее, один из переиспользуемых модулей - spring-web) - 
> дает и аннотацию `@RestController`, и почти все остальные аннотации, которые мы будем рассматривать в этой статье.

Первый вариант мы уже наблюдали ранее:

```java
@Controller
@RequestMapping("/hello")
public class HelloWorldController {
    @ResponseBody
    @GetMapping("/hello-world")
    public String hello() {
        return "Hello, World!";
    }
}
```

Класс аннотируется как `@Controller`, метод - аннотацией `@ResponseBody`. Дополнительно существуют аннотации для 
маппинга пути запроса, но о них поговорим чуть позже.

Тут может возникнуть логичный вопрос: если у нас за ответ сервера отвечает не напрямую объект `ServletResponse`, а 
метод с более-менее свободной сигнатурой - зачем нам вообще аннотация `@ResponseBody`? Если тела ответа нет - можно 
просто использовать `void`, если есть - возвращаемый тип, как мы и делаем сейчас.

Из этого вопроса растут ноги второй аннотации - `@RestController`.

Изначально Spring MVC предлагал в базовом сценарии возвращать из метода контроллера не тело, а внутренний адрес 
ресурса - JSP-страницы или чего-то еще. То есть стандартный, по мнению разработчиков, вариант выглядел бы так:

```java
@Controller
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping("/hello-world")
    public String hello() {
        return "/hello-world-page";
    }
}
```

Где `/hello-world-page` - некий внутренний идентификатор ресурса, чаще всего - JSP или HTML-страницы. То есть тело 
ответа с точки зрения HTTP есть - сама эта страница, но метод возвращает не ее, а ее идентификатор, который Spring 
MVC потом будет резолвить в своей внутренней логике - мы разберем это подробнее при знакомстве с Dispatcher Servlet.

Соответственно нужно было как-то разделить сценарии, когда возвращается идентификатор ресурса и некая DTO или иной 
объект, который в дальнейшем будет сериализован в JSON, XML или другой формат, в зависимости от настроек. Таким 
разделителем стала аннотация `@ResponseBody`.

Соответственно:

- `@Controller` над классом и `@ResponseBody`* над методом - метод контроллера возвращает объект, который надо 
  сериализовать в тело ответа;
- `@Controller` над классом БЕЗ `@ResponseBody` над методом - метод возвращает идентификатор ресурса, в тело ответа 
  надо записать зарезолвленный ресурс - скомпилированную JSP-страницу или что-то еще.

> *Аннотацию также можно указать над классом-контроллером и тогда она будет покрывать все методы этого класса:
> 
> ```java
> @Controller
> @ResponseBody
> @RequestMapping("/hello")
> public class HelloWorldController {
>   @GetMapping("/hello-world")
>   public String hello() {
>       return "Hello, World!";
>   }
> }
> ```

Но при этом довольно очевидно, что если мое приложение предоставляет именно API и всегда возвращает ответы в виде 
того же JSON - неудобно докидывать лишнюю аннотацию над каждым методом контроллера (или даже классом). Что делать? 
Правильно, ввести новую аннотацию, которая объединит `@Controller` и `@ResponseBody`. И назвать ее `@RestController`.

> Изначально аннотация, исходя из названия, предназначалась для реализации REST-парадигмы в Spring. О самой парадигме 
> поговорим в отдельной статье. Сейчас нас интересует то, что фактически аннотация может применяться для любой 
> HTTP-коммуникации, в том числе не соответствующей правилам REST. Скажем, REST регламентирует определенный набор 
> правил формирования путей, однако никто не запрещает пренебрегать ими в вашем приложении. Есть и ряд других 
> примеров, где фактическая семантика оказалась сильно шире декларируемой в названии.

С `@RestController` контроллеры стали чуть лаконичнее:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
  @GetMapping("/hello-world")
  public String hello() {
      return "Hello, World!";
  }
}
```

Более того, если мы захотим передать в теле ответа составной объект - Spring MVC внутри себя использует ряд внешних 
библиотек для конвертации, наиболее популярная из которых - FasterXML Jackson. Благодаря чему мы получаем практически 
бесшовную сериализацию в JSON (по умолчанию) или XML (при небольшой донастройке). Достаточно подключить в проект
Jackson и можно проверить на практике:

```groovy
    implementation 'com.fasterxml.jackson.core:jackson-databind:2.17.2'
```

```java
// Какой-то Java-класс, с понятными для Jackson правилами сериализации. Поля не существенны, просто пример с потолка
public class HelloWorldDto {
  // Старая дата используется, чтобы не погружаться в конфигурацию Jackson с поддержкой java.time
    private Date currentDate = new Date();
    private String appName = "HelloWorldApp";
    
    // ... - Геттеры. Здесь это важно, т.к. они нужны для автоматической десериализации через Jackson
}

@RestController
@RequestMapping("/hello")
public class HelloWorldController {
  @GetMapping("/hello-world")
  public HelloWorldDto hello() {
    return new HelloWorldDto();
  }
}
```

Теперь в ответ на запрос будет возвращаться примерно такой JSON-объект:

```json
{
  "currentDate": 1777314419617,
  "appName": "HelloWorldApp"
}
```

Как видите, для прикладного разработчика это в разы проще, чем делать аналогичную функциональность через Servlet API.

### Маппинг пути запроса

Отдельно нас интересуют аннотации, которые позволяют указать, какой путь (или пути) обрабатывает конкретный 
контроллер и отдельные методы этого контроллера. Какой метод обрабатывает конкретный метод контроллера, в более 
редких случаях может указываться набор еще некоторых параметров - подробнее рассмотрим ниже.

Речь об аннотации `@RequestMapping` и ее логических наследниках:

- `@GetMapping`;
- `@DeleteMapping`;
- `@PatchMapping`;
- `@PostMapping`;
- `@PutMapping`.

`@GetMapping` и остальные подобные - почти аналог `@RequestMapping`, но с четко заданным методом HTTP. То есть 
`GetMapping` предназначен строго для обработки GET-запросов и так далее. Также `@RequestMapping` может быть применен 
как над классом (для указания префикса пути, общего для всех методов контроллера), так и над методом контроллера.
`@GetMapping` сотоварищи может быть использован только над методом, что, в целом, логично.

> Несложно заметить, что `CONNECT`, `HEAD`, `OPTIONS` и `TRACE` не удостоились собственных методов. Вероятно, 
> аннотации заводились под те из методов, которые представлены в REST, но едва ли это имеет большое значение на 
> практике - как правило, даже 5 методов хватает с лихвой. Если не хватит - альтернативу рассмотрим ниже.

На практике чаще всего можно увидеть довольно простые конструкции в виде `@RequestMapping` с префиксом пути над 
классом и набор `@GetMapping`, `@PostMapping` и иже с ними аннотаций над методами - с уточнением путей или без них:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping // Обрабатывает GET-запросы по пути /hello
    public String basicGetHello() {
        // ... - тело метода
    }
    @PostMapping // Обрабатывает POST-запросы по пути /hello
    public String basicPostHello() {
        // ... - тело метода
    }
    
    @GetMapping("/hello-world") // Обрабатывает GET-запросы по пути /hello/hello-world
    public String getHello() {
        // ... - тело метода
    }
    
    @PostMapping("/hello-world") // Обрабатывает POST-запросы по пути /hello/hello-world
    public String postHello() {
        // ... - тело метода
    }
    
    @PostMapping("/hello-world/added-postfix") // Обрабатывает POST-запросы по пути /hello/hello-world/added-postfix
    public String postHello() {
        // ... - тело метода
    }
}
```

К слову, `/` в начале пути - штука опциональная, но обычно ее пишут. На поведение это не влияет, важны лишь 
промежуточные слэши в сложных путях:

```java
@RestController
@RequestMapping("hello") // Отсутствие слеша в начале пути не имеет значения
public class HelloWorldController {
    @PostMapping("hello-world") // Отсутствие слеша в начале пути не имеет значения
    public String postHello() {
        // ... - тело метода
    }
    
    @PostMapping("hello-world/added-postfix") // Отсутствие слеша в начале пути не имеет значения
    public String postHello() {
        // ... - тело метода
    }
}
```

Также в пути могут использоваться паттерны (`/my/path/**`), плейсхолдеры (`/my/path/${env.var.value}`) и даже 
SpEL-выражения (`/my/path/#{@myBeanName.anyStringMethod()}`), но такие возможности обычно не используются на практике.

Имеет смысл тезисно рассмотреть и другие атрибуты `@RequestMapping`, отметив наиболее популярные. Рассматривать 
отдельно атрибуты аннотаций, подобных `@GetMapping`, нет смысла, поскольку они лишь копия основной аннотации и 
проксируются в нее.

- `path` (алиас - `value`). Рассмотрен выше, наиболее популярный атрибут, в том или ином виде заполняется почти 
  всегда, как минимум над классом. Единственное, что стоит отметить, кроме уже описанного выше - принимает на вход 
  массив строк. Это крайне полезно при развитии системы - указывать несколько путей (сегментов путей), 
  относящихся к разным версиям API. Чаще всего в целях обратной совместимости;
- `name`. Логическое имя маппинга, доступное извне. Теоретически может быть полезно, чтобы по имени получать
  фактический путь (или сегмент пути), который этот маппинг описывает. Но чаще для тех же целей значение `path` 
  оформляют как константу. Не могу сказать, что этот атрибут сколь-либо популярен;
- `method`. Позволяет указать обрабатываемый метод (или методы) HTTP. Актуально при использовании аннотации 
  над методом*. Чаще всего закрывается аннотациями вроде `@GetMapping`. Может быть актуально либо при обработке одним 
  Java-методом нескольких методов HTTP (можно, но зачем?), либо при необходимости использовать метод, не имеющий 
  аннотации-алиаса. Тип атрибута - `RequestMethod`, через него доступны `HEAD`, `OPTIONS` и `TRACE` методы HTTP. 
  Впрочем, я с трудом представляю, зачем может потребоваться обработка `OPTIONS` или `TRACE` на уровне приложения, 
  `HEAD` - просто не популярен из-за своей семантики;
- `params`. Атрибут, позволяющий указать условия, по которым маппинг запроса будет учитывать не только путь, но и 
  определенные значения заданных параметров запроса. То есть запрос по конкретному пути может быть обработан тем или 
  иным методом контроллера, в зависимости от наличия определенных параметров запроса и их значений. На практике в 
  таких ситуациях чаще всего будет определено два разных пути с соответствующими методами контроллера, либо же 
  внутри самого метода контроллера будет описано ветвление логики. Крайне непопулярный атрибут, но имеющий право на 
  жизнь для legacy-систем на период рефакторинга;
- `headers`. Идея аналогична предыдущему атрибуту, но для заголовков запроса. Не популярно в современной разработке;
- `consumes`. Формально - аналог `headers`, но конкретно для заголовка `Content-Type`. Но на практике используется в 
  формате единственной реализации - для документации ожидаемого типа тела запроса и его валидации;
- `produces`. Аналогично предыдущему, но для заголовка `Accept`. Реальное использование - тоже в формате 
  единственной реализации.

> *Технически можно использовать данный атрибут и при аннотировании класса. Тогда это будет в первую очередь 
> ограничением для методов (иные методы HTTP использовать нельзя). Ну, или все указанные методы HTTP могут быть 
> обработаны соответствующими Java-методами, если для конкретного Java-метода не указано иное.
> 
> В любом случае, не могу вспомнить использования этой возможности на практике.

Для `consumes` и `produces` хочется привести пример. Изначально они, вероятно, предполагали, что по одному пути 
может быть доступна коммуникация как с телом запроса (или ответа) в виде, скажем JSON, так и в виде XML. И 
обрабатывать это предполагалось разными методами контроллера.

Но более реалистичный сценарий - приложение ожидает только один формат (допустим, JSON), а все остальное должно 
отвечать общей ошибкой 415 (Unsupported Media Type):

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @PostMapping(
            value = "/hello-world",
            consumes = MediaType.APPLICATION_JSON_VALUE,
            produces = MediaType.APPLICATION_JSON_VALUE
    )
    // @RequestBody - аннотация для работы с телом запроса, рассматриваем чуть ниже, здесь особого значения не имеет
    public HelloWorldDto hello(@RequestBody Object helloDto) {
        // ... - тело метода
    }
}
```

Корректный запрос:

```shell
curl -X POST --location "http:// localhost:8080/ hello/ hello-world" 
  \ -H "Content-Type: application/ json"
  \ -H "Accept: application/ json"
  \ -d '{ "id": 999, "value": "content" }'
```

Некорректный (в теле и `Content-Type` - XML) запрос:

```shell
curl -X POST --location "http:// localhost:8080/ hello/ hello-world" 
  \ -H "Content-Type: application/ xml"
  \ -H "Accept: application/ json"
  \ -d '<any-tag>any value</any-tag>'
```

Во втором случае Spring не будет даже пытаться читать тело - убедившись, что значение в хедере отлично от ожидаемого,
он сразу выбросит 415 ошибку. 

## Параметры метода контроллера

Данный пункт описывает основные способы вытянуть в метод контроллера информацию, содержащуюся в запросе. Фактически 
мы будем говорить о том, что может стать параметром метода контроллера, обрабатывающего HTTP-запрос*.

> *Ниже мы увидим, что не все методы контроллера предназначены для обработки HTTP-запросов

### Сегменты пути

На мой взгляд, одна из наиболее революционных концепций Spring MVC относительно Servlet API - возможность 
использовать части пути как полезную нагрузку запроса. Это напрямую следует из REST (разберем позже), но после 
Servlet API с довольно жестким маппингом пути на сервлет выглядит ново:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    // Еще один допустимый формат сегмента пути - имя переменной пути внутри фигурных скобок
    @GetMapping(value = "/hello-world/{myPathVar}")
    // Значение из сегмента, эквивалентного {myPathVar} будет записано в параметр метода.
    // То есть для запроса по пути /hello/hello-world/1 в параметр метода "myPathVar" будет записано "1", а
    // для запроса по пути /hello/hello-world/qwerty в параметр myPathVar запишется "qwerty"
    public HelloWorldDto hello(@PathVariable String myPathVar) {
        // ... - тело метода
    }
}
```

Параметры, в которые будет мапиться сегмент пути, могут принимать разные типы данных: строку, `UUID`, целочисленные 
типы... Если значение не сможет быть конвертировано целевой тип - возникнет исключение. Обычно это не является 
проблемой, поскольку в реальных приложениях пути запросов согласуются заранее и такие динамические path variable 
тоже должны заполняться клиентом в соответствии в согласованным контрактом коммуникации.

Пример выше - наиболее лаконичный. Первое, что стоит понять - как происходит биндинг переменной в пути и параметра 
метода. Опции две:

1. Имя переменной в пути (то, что обрамлено фигурными скобками) и имя параметра в методе совпадают. Как в примере 
   выше. В таком случае запись значения в параметр произойдет автоматически*;
2. Если имена по какой-то причине должны отличаться - можно использовать атрибуты аннотации - `name` или `value`. 
   Пример ниже.

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping(value = "/hello-world/{myPathVar}")
    public HelloWorldDto hello(@PathVariable("myPathVar") String anotherParameterName) {
        // ... - тело метода
    }
}
```

> *Для корректной работоспособности может потребоваться донастройка компилятора - обеспечение доступности имен 
> параметров в методе через рефлексию. Решается одним флагом в настройках компилятора (можно задать через систему 
> сборки).
> 
> Честно говоря, не помню такого в своей практике, но наткнулся на проблему буквально при написании этой статьи. И как 
> показывает поиск по Сети, проблема не уникальна. Если столкнулись с чем-то похожим - смотрите в сторону флага 
> `-parameters` для компилятора. В Gradle можно установить, добавив в `build.gradle`:
> 
> ```groovy
> tasks.withType(JavaCompile).configureEach {
>     options.compilerArgs += ['-parameters']
> }
> ```

Разумеется, можно использовать несколько path variable в одном маппинге. Скажем, путь маппинга может быть 
представлен как `/prefix/{firstVar}/intermediate-segment/{secondVar}`. В методе контроллера при этом будет два 
параметра - один для `firstVar`, второй - для `secondVar`.

Также у `@PathVariable` есть непопулярный атрибут - `required`. Он указывает, обязательно ли должна быть заполнена 
переменная. Отсюда может следовать заблуждение, что переменную пути можно как-то оставить пустой, только непонятно, 
как должен выглядеть подобный URL.

Такой флаг может иметь смысл, если метод контроллера обрабатывает несколько путей, и лишь в части из них есть нужная 
переменная:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping(value = {"/with-path-var/{myPathVar}", "/without/path/var"})
    // При обработке методом второго пути значение myPathVar будет null. Также можно сам параметр метода обернуть в 
    // Optional - этот подход также допустим
    public HelloWorldDto hello(@PathVariable(required = false) String myPathVar) {
        // ... - тело метода
    }
}
```

Позже, знакомясь с REST API, мы рассмотрим практику применения `@PathVariable` - какие данные и в каких ситуациях 
рекомендуется передавать как часть пути, а не переменные запроса или тело. 

### Параметры запроса

Работа с параметрами запроса может выглядеть по-разному. Наиболее просто и популярный вариант рассмотрим в этом 
пункте, более продвинутый - ниже, среди legacy-функциональности. Возможно, он там оказался не совсем заслуженно, но 
иначе структура статьи страдала еще больше:)

Итак, работа с параметрами запроса. Фактически, это обертка над `ServletRequest#getParameter()` и 
`ServletRequest#getParameterValues()`:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    // Допустим, обрабатываем запрос /hello/hello-world?param1=qwe&param2=2&param3=rty&param3=uio&param4=asdf
    @GetMapping("/hello-world")
    public HelloWorldDto hello(
            // Аннотация @RequestParam позволяет биндить параметры запроса на параметры метода-обработчика.
            @RequestParam String param1,
            @RequestParam Integer param2,
            // Доступны и единичные значения с их автоматической конвертацией в строку, целочисленный тип, элемент 
            // заданного enum'а, boolean или UUID. Также есть поддержка типов даты и времени, можно добавить и свои 
            // правила конвертации входящих значений 
            @RequestParam List<String> param3,
            // И множественные значения, которые можно оформить в виде коллекции. Элементы могут быть тех же типов, 
            // которые доступны для единичных значений 
            @RequestParam Set<String> param4
    ) {
        // ... - тело метода
    }
}
```

Идея параметров запроса достаточно примитивна и в данном случае здесь нет ничего кардинально нового, относительно 
Servlet API.

Единственное, стоит обратить внимание на атрибуты аннотации `@RequestParam`:

- `value` и `name`. Взаимозаменяемы. Логика та же, что и в `@PathVariable` - если имя параметра в URI отличается от 
  имени параметра в методе - их можно сбиндить через этот атрибут;
- `required`. Опять же, обязательность параметра. Здесь поведение более очевидно - параметр может быть передан в 
  запрос, а может и нет. По умолчанию считается, что параметр обязательный;
- `defaultValue`. Позволяет установить строковый эквивалент значения по умолчанию. Если фактический параметр метода 
  не строка - значение автоматически конвертируется в нужный тип. Используется, если параметр не задан в URI. Сам 
  факт заполнения этого атрибута переключает `required` в значение `false`.

### Заголовки запроса

Работа с заголовками на уровне контроллера требуется относительно редко - обычно заголовки несут либо 
инфраструктурную нагрузку (тип контента, размер тела, значения для CORS-политик и т.д.), либо сквозную - сессии, 
токены и другие элементы, которые логичнее обработать фильтром или его аналогом, нежели методом контроллера. Для 
аналогии можно вспомнить, что в сервлетных приложениях мы также обычно не обращались к хедерам внутри самих 
классов-сервлетов.

Тем не менее, такая функциональность есть и время от времени она становится полезна.

Мы рассмотрим две наиболее популярные аннотации - на самом деле их больше, но часть из них слишком специфична, еще 
часть - находится за пределами Spring MVC.

1. `@RequestHeader`. Фактически - аналог `HttpServletRequest#getHeader()` и родственных ему методов. По атрибутам и 
   их семантике аннотация полностью аналогична `@RequestParam`, поэтому разбирать атрибуты отдельно не имеет смысла; 
2. `@CookieValue`. Более специфичная аннотация для Cookies. Обертка над `HttpServletRequest#getCookies()`, 
   атрибутивный состав и его семантика аналогична `@RequestHeader` и `@RequestParam`.

Рассмотрим пример:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping("/hello-world")
    public HelloWorldDto hello(
            // Допустим, хотим получить информацию о браузере пользователя. Очевидно, для заголовков использовать 
            // явный биндинг по имени актуальнее, нежели для path variable и параметров запроса
            @RequestHeader("User-Agent") String userAgent,
            // Логика конвертации значений для заголовков и cookies та же, что и для параметров запроса.
            @CookieValue("MY_CUSTOM_COOKIE_1") Integer myCookie,
            // Именно для cookies можно использовать также тип Cookie из Servlet API, если нужны метаданные.
            // Само значение можно получить сразу указав его целевой тип, как это сделано строкой выше
            @CookieValue("MY_CUSTOM_COOKIE_2") Cookie myCookieWithMetaData
    ) {
        // ... - тело метода
    }
}
```

### Тело запроса

Остается разобрать самую интересную часть - тело запроса.

В большинстве случаев разработчик взаимодействует с бинарными или строковыми форматами, которые десериализуются в 
обычные DTO - JSON (наиболее вероятно в сегодняшних реалиях), XML и так далее.

Такие данные обычно обрабатываются через аннотацию `@RequestBody` - она буквально означает, что тело запроса нужно 
представить в виде DTO указанного типа. Если не говорить о каких-то необычных настройках и кастомизациях, за процесс 
десериализации будет отвечать хорошо нам знакомый FasterXML Jackson*.

> *Как минимум это означает, что если формат DTO не сформирован или заранее не ясен - можно использовать `Object`,
> `Map<String, Object`, `List<Object>` и другие комбинации стандартных для Jackson типов - в зависимости от текущих
> потребностей конкретного запроса.
> 
> Как максимум - JSON будет автоматически десериализоваться в правильно описанную DTO (хотя бы имеющую геттеры). 
> Если вспомнить, насколько сложнее было реализовать схожую логику на голом сервлетном приложени - это огромный шаг 
> вперед.

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @PostMapping("/hello-world/v1")
    public HelloWorldDto hello(@RequestBody Object helloDto) {
        // ... - тело метода
    }
    
    @PostMapping("/hello-world/v2")
    public HelloWorldDto hello(@RequestBody HelloWorldDto helloDto) {
        // ... - тело метода
    }
    
    @PostMapping("/hello-world/v3")
    public HelloWorldDto hello(@RequestBody List<HelloWorldDto> helloDtos) {
        // ... - тело метода
    }
}
```

Также изредка нам может потребоваться что-то совсем не структурированное - массив байт, например*. Можно либо 
использовать `ServletRequest` напрямую - там доступен и вариант с `InputStream`, и `Reader` для него же. Но можно 
использовать и `@RequestBody`:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @PostMapping("/hello-world/v4")
    public HelloWorldDto hello(@RequestBody byte[] helloBytes) {
        // ... - тело метода
    }
    
    @PostMapping("/hello-world/v5")
    public HelloWorldDto hello(@RequestBody InputStream helloStream) {
        // ... - тело метода
    }
}
```

> *Главное отделять такие ситуации от передачи файлов и иных случаев использования `multipart/form-data` - там это 
> будет работать, но крайне неудобно. Более корректный способ описан ниже.

Вообще, `@RequestBody` довольно проста по своему использованию. Все возможные подводные камни кроются в конфигурации 
Spring MVC или непосредственно используемого им `ObjectMapper`, но это не тема сегодняшней статьи.

Сама аннотация имеет единственный атрибут - `required`. Собственно, указывает, может ли запрос прийти без тела. 
Значение по умолчанию - `true`, что означает, что тело должно быть, пусть даже в виде пустого JSON.

Больше ограничений может возникать исходя из семантики. Во-первых, тело запроса может быть только одно - в одном 
методе контроллера не может быть несколько параметров, аннотированных `@RequestBody`.

Во-вторых - семантика HTTP. Как и сервлетные приложения в целом, Spring MVC не ограничивает использование тела запроса
для любых методов HTTP. Однако браузеры, прокси сервера и другие элементы инфраструктуры в клиент-серверной коммуникации
могут иметь собственное мнение на этот счет. Из-за чего не стоит без особых причин использовать `@RequestBody` (и тело
запроса в принципе) с `GET` и `DELETE` методами.

### Передача мультимедиа

Что же с файлами? Допустим, мы хотим загрузить на сервер аудио, видео или просто изображение.

Тема отправки подобных данных через HTTP имеет свою подноготную - как минимум в части заполнения `Content-Type`, а 
также возможности отправлять несколько файлов или файл и какую-то информацию помимо него в одном запросе.

В конечном итоге, зачастую отправка файлов будет идти рука об руку с `Content-Type` со значением 
`multipart/form-data`. Фактически это означает многосоставное тело запроса, но это отдельная большая тема со своим 
историческим контекстом. Поэтому просто зафиксируем такое значение заголовка, оно будет иметь значение далее.

Второй нюанс, который стоит помнить при работе с файлами - семантику метода `ServletRequest#getParameter()`. Напомню,
он работает с parameter map и покрывает не только с параметрами запроса, но и, например, данные url-encoded form. 
Важное для нас следствие - раз это не разделяется на уровне объекта сервлетного запроса, это не может быть разделено 
и на уровне `@RequestParam`.

Наконец, третий нюанс. Работа с файлами (а точнее, с `multipart/form-data`) требует дополнительной конфигурации 
сервлета. В наших реалиях - dispatcher servlet'а. Есть несколько способов это сделать, мы пойдем по самому простому 
пути.

В прошлой статье мы создавали класс `WebAppInitializer`, наследника ~~Исильдура~~ 
`AbstractAnnotationConfigDispatcherServletInitializer`. Это и есть конфигурация dispatcher servlet. В нашем случае 
стоит переопределить один из методов в этом классе:

```java
public class WebAppInitializer extends AbstractAnnotationConfigDispatcherServletInitializer {
    // ... - ранее описанные методы

    @Override
    protected void customizeRegistration(ServletRegistration.Dynamic registration) {
        // Необходимо указать настройки для multipart - в первую очередь, максимально допустимые размеры 
        registration.setMultipartConfig(
            new MultipartConfigElement(
                // Путь для сохранения файлов на сервере. В современных реалиях файлы обычно не хранятся на серверах с
                // приложением, не считая временных файлов, удаляемых после обработки запроса
                null,
                // Максимальный размер для файла в байтах
                100_000_000,
                // Максимальный размер запроса в байтах целиком. Может содержать не только один файл
                20_000_000,
                // Размер буфера в байтах. Все, что больше - пишется на диск
                1_000_000       
            )
        );
    }
}
```

Также для корректной работы требуется бин типа `MultipartResolver` - в классической конфигурации достаточно чего-то 
в стиле

```java
@Bean
public MultipartResolver multipartResolver() {
    return new StandardServletMultipartResolver();
}
```

> Отдельно отмечу, что конфиги выше - в первую очередь для классических Spring MVC приложений. В Spring Boot часть 
> этого вынесена в автоконфигурацию, отдельные конфиги доступны для управления через application-файл в более 
> комфортном режиме.

Детали выше - скорее шпаргалка для тех, кому придется оперативно решать задачу загрузки файла на сервер или же 
просто нужны первые зацепки для погружения в тему. Просто так тратить время на погружение в специфику передачи 
файлов по HTTP мне кажется неоправданным, поэтому зазубривать или глубоко вникать в детали выше прямо сейчас, вероятно,
не стоит.

Теперь к нашим реалиям и API. Есть два основных способа загрузить файл на сервер через метод контроллера.

Первый из них работает через уже известную нам аннотацию `@RequestParam`:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @PostMapping("/hello-world/upload/v1")
    // Имя параметра file - условно. Оно определяется контрактом конкретного эндпоинта и может быть любым.
    // Класс MultipartFile - тип Spring MVC, позволяющий получить основную информацию о файле: тип, размер, имя, 
    // содержимое и базовые методы для конвертации в File или Path для дальнейшей обработки через java.io или java.nio
    public void upload1(@RequestParam("file") MultipartFile file) {
        // ... - тело метода
    }
    
    @PostMapping("/hello-world/upload/v2")
    // Более популярный формат, именно в данном примере особых семантических отличий от предыдущего нет. Атрибуты 
    // аннотации @RequestPart стандартны: value/name, required.
    public void upload2(@RequestPart("file") MultipartFile file) {
        // ... - тело метода
    }
    
    @PostMapping("/hello-world/upload/v3")
    // Не рекомендую делать так на практике, но пример расхожий - технически, можно передать и файл, и какой-нибудь 
    // JSON одним запросом. При правильной организации внутри тела запроса, это будет корректно обработано - и 
    // сформируется MultipartFile с нужными метаданными, и автоматически десериализуется JSON в соответствующую DTO. 
    // Суть в том, что в отличие от @RequestBody, @RequestPart изначально подразумевает работу с частью тела 
    // multipart-запроса, соответственно, частей может быть несколько.
    //
    // Как и в примерах выше, используемые имена (file, dto) условны и определяются контрактом конкретного эндпоинта.
    public void upload2(@RequestPart("file") MultipartFile file, @RequestPart("dto") MyDto dto) {
        // ... - тело метода
    }
}
```

В одном запросе также может передаваться несколько файлов в виде списка (в Java-отображении), в виде отдельных 
request part и множество иных вариаций. Но это еще более узкие задачи, в описание которых не хочется углубляться. 
Основная цель данного подпункта - расширение вариантов использования `@RequestParam` и знакомство с `@RequestPart`.

### `ServletRequest` и `ServletResponse`

Если какой-то функциональности не хватило из методов выше, всегда можно воспользоваться сервлетным стилем: указать 
в параметрах запроса `ServletRequest`/`HttpServletRequest` и/или `ServletResponse`/`HttpServletResponse`. Последняя 
опция несколько выбивается из основной темы пункта, но все еще относится к возможным параметрам метода контроллера.

Получив сервлетные объекты запроса или ответа можно обрабатывать их в виде, привычном для сервлетного приложения. 
Однако на практике такой подход используется нечасто - как правило, достаточно аннотаций, описанных выше. А для 
каких-то более узких случаев - менее популярных аннотаций из Spring MVC или смежных модулей Spring.

> В моей практике встречалось всего несколько ситуаций, когда в метод контроллера явно передавали `HttpServletRequest`.
> И эти ситуации будет честнее объяснить историческими причинами или костылями на следующих слоях архитектуры, 
> нежели реальной необходимостью или удобством. 

Пример:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping("/hello-world")
    // Объект запроса передаем - вероятно, все, что захотим, можем достать из него. А ответ все еще формируем через 
    // DTO, которое Spring сериализует и запишет в ответ самостоятельно.
    // Допустимый вариант, но субъективно не элегантный.
    // 
    // Для передачи объекта запроса или ответа дополнительные аннотации не нужны. 
    public HelloWorldDto hello(HttpServletRequest request) {
        // ... - тело метода
    }
}
```

## Формирование ответа метода контроллера

### `@ResponseBody` и `ResponseEntity`

Сама аннотация фактически была объяснена еще в начале статьи - она нужна, чтобы сепарировать подход, основанный на 
обращении к ресурсу от подхода, предполагающего формирование и отправку полезной нагрузки в виде сериализуемого объекта.

Однако стоит отметить, что Spring MVC предлагает еще один способ оформления тела (и не только тела) ответа - класс 
`ResponseEntity`.

`ResponseEntity` предлагает несколько разнообразных конструкторов и методов для формирования успешного или 
ошибочного ответа с телом, конфигурацией заголовков и HTTP Status Code:

```java
// Практика использовать именно @RestController закрепилась, но если заменить ее на @Controller - для методов, 
// возвращающих ResponseEntity<?> ничего не изменится - этот тип имеет особый флоу обработки
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping("/hello-world/success")
    public ResponseEntity<HelloWorldDto> helloSuccess() {
        return ResponseEntity.ok(new HelloWorldDto()); // устанавливает 200 код и тело (параметром)
    }

    @GetMapping("/hello-world/failed")
    public ResponseEntity<HelloWorldDto> helloFailed() {
        return ResponseEntity.status(500) // Формирование через builder
                .header("My-Header-Name", "myHeaderValue")
                .body(new HelloWorldDto());
    }
}
```

По сути, `ResponseEntity` - это тип-контейнер, хранящий внутри себя тело, заголовки и статус ответа. Дополнительно к 
этому разработчики Spring предусмотрели весьма гибкий набор способов создания объекта - конструкторы, статические 
методы и builder. Детально разбирать API не будем, он интуитивно понятен не выходя за пределы автодополнения IDEA.

Стоит отметить, что в реальном проекте методы контроллеров чаще всего будут возвращать именно `ResponseEntity`. В 
этой статье данный класс используется минимально исключительно для того, чтобы не замыливать взгляд среди и так 
большого количества новых обозначений. В дальнейших статьях он будет применяться повсеместно. 

### @ResponseStatus

`@ResponseStatus` - аннотация, управляющая статусом ответа. Имеет достаточно много вариантов применения:

Над собственным классом исключения:

```java
@ResponseStatus(HttpStatus.BAD_REQUEST) // 400 код. Использован исключительно для примера
public class MyException extends RuntimeException {
    
}
```

Если контроллер выбросит `MyException` и не будет каких-то дополнительных обработчиков ошибок - в ответ сервера 
запишется именно код, указанный в `@ResponseStatus`. Если будет выброшено никем не обработанное исключение без четко 
заданного кода - как и в любом сервлетном приложении кодом ошибки по умолчанию является 500.

Также `@ResponseStatus` может использоваться над методом контроллера (в том числе для указания успешного кода, 
отличного от 200):

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @ResponseStatus(HttpStatus.CREATED) // 201
    @PostMapping("/hello-world/success")
    public HelloWorldDto helloSuccess() {
        // ... - тело метода
    }
}
```

Третий вариант применений - `@ExceptionHandler`-метод. Он будет разобран в завершающем пункте статьи, пример с 
`@ResponseStatus` можно найти там же.

## Legacy-инструменты

Этот пункт статьи - преимущественно тезисный. Он описывает аннотации, которые актуальны при работе с приложениями, 
использующими JSP или иные шаблонизаторы. Что, в свою очередь, практически не актуально в современных приложениях.

### `@ModelAttribute`

Фактически, это надстройка над `ServletRequest#getParameterMap()`, которая в наиболее популярном сценарии разбирает 
параметры запроса или содержимое URL-encoded form в заданный тип DTO. В такой парадигме это похоже на аналог 
`@RequestBody`*:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping("/hello-world")
    // Содержимое Parameter Map вписалось в MyDto по совпадающим ключам
    public HelloWorldDto hello(@ModelAttribute MyDto dto) {
        // ... - Логика обработки запроса
    }
}
```

> *Не стоит сильно привыкать к подобным аналогиям - `@RequestBody` и `@ModelAttribute` базируются на очень разных
> инструментах из Servlet API и очень по-разному обрабатываются в Spring MVC. Но как наиболее простая ассоциация
> на первых этапах знакомства может иметь смысл.

> Заметка на полях.
> Тут, казалось бы, впору заметить, что URL-encoded формы сами по себе устарели и не используются в современной 
> разработке. Однако на одном из моих последних проектов url-encoded-form оказалось в требованиях к server-server 
> интеграции - этот формат в принципе для сервера подходит довольно слабо, однако бывает и так.
> 
> Собственно, именно через @ModelAttribute на своей стороне мы такие запросы и обрабатывали.

Также стоит отметить, что аннотация `@ModelAttribute`, на самом деле, опциональна. То есть это хороший маркер и если 
это не единственный параметр метода контроллера - ее лучше ставить. Но если все, что принимает метод контроллера - 
это `ServletRequest#getParameterMap()`, оформленный в виде DTO - можно убрать аннотацию и поведение останется тем же:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @GetMapping("/hello-world")
    public HelloWorldDto hello(MyDto dto) {
        // ... - Логика обработки запроса
    }
}
```

В таком виде конструкция даже более популярна, чем с `@ModelAttribute`. Обычно так оформляют методы с большим 
количеством параметров запроса - лучше одна DTO, нежели много параметров, обозначенных как `@RequestParam`. В виде 
DTO зачастую проще обрабатывать в дальнейшей логике, проще описывать правила валидации и т.д.

Однако вернемся на секунду к `@ModelAttribute`. Эта аннотация может применяться не только к параметрам, но и к методам. 
И в таком случае придется разбираться, что значит "Model" в названии и как это связано с моделью в паттерне MVC. А уже
это находится за пределами текущей статьи и, в общем, за пределами современных реалий Java-разработки. 

### `@SessionAttribute`

Если `@ModelAttribute` был логической надстройкой поверх `ServletRequest#getParameterMap()`, то `@SessionAttribute` - 
надстройка и доработка вокруг `HttpSession`.

Логика использования аннотации - фактически получение атрибута сессии, более удобная форма цепочки 
`HttpServletRequest#getSession()` -> `HttpSession#getAttribute()`:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    @PostMapping("/hello-world")
    // В отличие от сервлетного API здесь Spring сам сделает приведение к нужному типу. 
    public HelloWorldDto hello(@SessionAttribute("attributeName") MyAttributeType attribute) {
        // ... - Логика обработки запроса
    }
}
```

Добавление атрибутов возможно либо через класс модели (не рассматривали в статье), либо через стандартный Servlet API.

Современные приложения часто отходят от сессионной аутентификации в сторону token-based форматов - мы еще вернемся к 
этой теме при изучении Spring Security. Однако сессии все еще имеют право на жизнь, особенно в простых приложениях. 
Соответственно, время от времени мы будем использовать схожие механизмы, но не на слое контроллеров.

### `@SessionAttributes`

`@SessionAttributes` - более специфичный инструмент, позволяющий привязывать определенные атрибуты к контроллеру и
сохранять их значения (в пределах одной HTTP-сессии) именно в разрезе данного контроллера. Это актуально для ситуаций,
когда какая-то форма имеет несколько состояний и необходимо хранить контекст, пока пользователь пройдет все их целиком.
То есть фактически это сохранение определенных данных не как атрибута сессии в принципе, а в пределах сессии и
контроллера.

Рассматривать пример не будем - если остальные легаси-аннотации еще имеют какой-то смысл - или как концепция, или с 
хотя бы побочным применением в сегодняшних реалиях, то `@SessionAttributes` - специализированный инструмент под узкую
задачу.

## Биндинги и обработка ошибок

Еще один пункт, который на данном этапе рассмотрим совсем тезисно - более подробно будем рассматривать предложенные 
аннотации в разрезе статьей об обработке ошибок и валидации.

### `@InitBinder`

Большее значение эта аннотация имела в legacy-подходе с использованием ресурсов - фактически, она отвечала за 
львиную долю конвертаций и форматирования данных, отчасти за настройки безопасности в моделях. 

В современных реалиях она полезна в узких сценариях. Скажем, регистрации самописных валидаторов для тела запроса для 
методов конкретного контроллера или группы контроллеров, в зависимости от места использования аннотации. Именно этот 
пример мы рассмотрим в соответствующей статье.

### `@ExceptionHandler`

Аннотация, позволяющая создавать метод-обработчики ошибок. Они способны отлавливать ошибку определенного типа и 
формировать ответ сервера на основании этой ошибки. Может применяться для методов контроллера, либо, чаще, для 
`@ControllerAdvice` (см. ниже).

Вероятно, основной способ обработки ошибок в Spring MVC, если не считать классический try-catch.

Выглядит это примерно так:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    // ... - Методы обработки запросов
    
    // MyCustomException - некое бизнес-исключение в нашем приложении
    @ExceptionHandler(MyCustomException.class)
    // ErrorResponse - некий тип, который, допустим, определен в нашем приложении и используется именно для ответов 
    // в случаях ошибки
    public ErrorResponse handleMyCustomException(MyCustomException ex) {
        // ... - Формирование ответа именно с учетом исключения MyCustomException
    }

    
    @ExceptionHandler(Exception.class)
    public ErrorResponse handleException(Exception ex) {
      // ... - Формирование ответа для любых иных исключений
    }
}
```

Теперь любая ошибка, выброшенная методом контроллера `HelloWorldController` будет обработана одним из двух 
методов-обработчиков - в зависимости от типа. Также такие методы умеют взаимодействовать с аннотацией 
`@ResponseStatus`. Наравне с `ResponseEntity`, она является одним из стандартных способов установить определенный 
статус код в `@ExceptionHandler`-методе:

```java
@RestController
@RequestMapping("/hello")
public class HelloWorldController {
    // 409 - код выбран исключительно для примера, в реальном мире стоит следовать семантике HTTP
    @ResponseStatus(HttpStatus.CONFLICT)
    @ExceptionHandler(MyCustomException.class)
    public ErrorResponse handleMyCustomException(MyCustomException ex) {
        return getErrorResponse();
    }

    
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleException(Exception ex) {
        ErrorResponse response = getErrorResponse();
        
        return new ResponseEntity(response, HttpStatus.METHOD_NOT_ALLOWED); // 405 
    }
    
    // ... - реализация getErrorResponse() для примеров выше
}
```

### @ControllerAdvice и @RestControllerAdvice

Эти аннотации почти всегда рассматриваются в парадигме глобальной обработки ошибок. Если вам хочется описать общие 
для всего приложения `@ExceptionHandler`'ы - создайте класс, аннотированный `@ControllerAdvice` (или 
`@RestControllerAdvice`) и опишите их там.

Реальность несколько шире. Обе аннотации позволяют создавать классы, в которых можно располагать методы, общие для 
всех (или группы) контроллеров. Если не учитывать варианты использования, потерявшие свою актуальность, остается как 
минимум:

1. Место для создания `@InitBinder`-метода. Не отдельного под каждый контроллер, а более глобального;
2. Собственно, обработка ошибок через `@ExceptionHandler`. Наиболее популярный вариант - зачастую можно увидеть, что 
   классы, аннотированные `@ControllerAdvice`, называются в стиле "`GlobalExceptionHandler`";
3. Имплементация `ResponseBodyAdvice` и `RequestBodyAdvice` без императивной регистрации в конфигурации. В целом, 
   эти механизмы находятся за пределами курса, в силу своей слабой популярности, но возможно мы затронем их при 
   рассмотрении Dispatcher Servlet.

Если сильно утрировать, класс, аннотированный `@ControllerAdvice` можно рассматривать как некую альтернативу 
введению суперкласса для всех контроллеров, где определены методы, нужные всем наследникам. В ряде случаев это будет 
ложной аналогией, но для наиболее общих сценариев семантика очень похожа.

Отличие между `@ControllerAdvice` и `@RestControllerAdvice` то же, что и у `@Controller`/`@RestController`: 
`@RestControllerAdvice` = `@ControllerAdvice` + `@ResponseBody`. Это актуально для `@ExceptionHandler`-методов, 
логика та же, что и с методами-обработчиками запросов в контроллерах. Аналогично с ними же, разница аннотаций не 
критична, если метод возвращает `ResponseEntity<?>`.

Также имеет смысл ознакомиться с атрибутами аннотации `@ControllerAdvice`, но этот разбор лучше отложить до статьи, 
посвященной обработке исключений - там удастся привести более наглядные примеры для некоторых из атрибутов.

## Заключение

По сути, все, что связано с самой логикой обработки запроса в Spring MVC можно разбить на две составляющие:

1. Как запрос обрабатывается с помощью dispatcher servlet. Это описывает внутренний путь запроса, именно то, что 
   добавляет Spring поверх само сервлетное приложение. Эту часть мы практически не затрагивали и ей посвящены 
   следующие статьи раздела;
2. Как можно взаимодействовать с объектами запроса и ответа - как и откуда читать, как и куда писать. 
   Преимущественно этому посвящена текущая статья, но также мы посмотрим на изученный инструментарий снова через 
   призму dispatcher servlet, который и создает всю магию в сегодняшних примерах.

В обоих случаях нет большого смысла в зазубривании каждого шага обработки запроса, каждой аннотации или параметра в
методе контроллера. Главное улавливать общую схему взаимодействия, а также знать, что в принципе можно извлечь из 
объекта сервлетного запроса и как можно взаимодействовать с объектом сервлетного ответа. Вкупе с общими 
представлениями о подходах Spring MVC это почти всегда позволит понять, где искать узкий инструмент под текущую 
задачу или же куда добавить нужную кастомную логику, чтобы все заработало.

Как и большинство других модулей Spring, Spring MVC с одной стороны - весьма логичен и последователен в именах, 
возможностях и ограничениях. С другой - усложнен собственным историческим контекстом, из-за чего отдельные вещи 
могут быть сделаны сложнее, чем хотелось бы или находятся в не самых очевидных местах.
