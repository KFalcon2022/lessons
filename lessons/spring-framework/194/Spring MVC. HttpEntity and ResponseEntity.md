# Spring MVC. `HttpEntity` и `ResponseEntity`

Сегодня рассмотрим API и семантику двух небольших классов - `HttpEntity` и `ResponseEntity`.

В их поведении нет ничего сложного, но с их использованием можно столкнуться в относительно большом перечне задач: при
возвращении ответа из метода контроллера, при обработке ошибок, при отправке запросов со стороны сервера и обработке их
ответов.

Отчасти мы уже сталкивались с ними в предыдущих статьях и неоднократно столкнемся в следующих. Эта статья позволит 
не отвлекаться на детали реализации и API в дальнейшем, сгруппировав все необходимое здесь.

## Актуальность

`HttpEntity` - сущность, описывающая тело и заголовки запроса или ответа. Начинающие бэкенд-разработчики зачастую 
считают, что их основная задача - работа с телом запроса или ответа. Однако в заголовках часто передаются не только 
технические метаданные вроде типа контента и его объема, но и множество вещей, затрагивающих бизнес-логику и
безопасность. И в таких ситуациях `HttpEntity` становится крайне удобным типом, позволяющим опираться на один общий 
объект вместо нескольких разрозненных, описывающих разные компоненты запроса (или ответа). 

Например, в заголовках может передаваться информация об авторизации в виде закодированных логина и пароля, токена
или идентификатора сессии (в Cookies). Также в заголовках может передаваться "подпись" запроса (например, 
HMAC) или иные данные.

Так или иначе, при разработке достаточно часто возникает ситуация, когда необходимо работать не только с телом 
запроса (или ответа), но также обрабатывать и заголовки. При обработке запросов речь будет идти о чтении заголовков 
запроса и записи заголовков ответа, при отправке запросов - наоборот. А раз с чем-то часто приходится работать 
вместе - почему бы не объединить эти данные в общую модель с удобным API.

Именно так и появился класс `HttpEntity`.

`ResponseEntity` - наследник `HttpEntity`. Отличается он в первую очередь тем, что заточен на работу с ответом и как 
проявление этого - предоставляет API для работы со статус-кодом ответа. Это удобно и для ситуаций формирования 
ответа на запрос, и в случаях, когда мы разбираемся с ответом, пришедшим на отправленный нами запрос - эту ситуацию 
мы рассмотрим подробнее в одной из следующих статей.

Таким образом, `HttpEntity` и `ResponseEntity` представляют собой высокоуровневые сущности, предоставляющие удобные 
API. Технически, все то же самое можно сделать с сервлетными объектами запроса и ответа напрямую, но куда менее 
удобно и с большим количеством бойлер-плейта. Этот пункт станет нагляднее при знакомстве с API.

## `HttpEntity` API

`HttpEntity` мало чем отличается от простой DTO с набором геттеров. Фактически этот класс предоставляет API для
управления двумя полями:

1. `HttpHeaders headers`. Высокоуровневая обертка для управления заголовками на основе `MultiValueMap`. 
   `MultiValueMap` - структура данных, предоставляемая Spring, позволяющая хранить список значений по ключу. В 
   случае с заголовками это полезно, поскольку один заголовок может содержать несколько значений. `HttpHeaders` 
   предоставляет как методы, классические для `Map` и `MultiValueMap` (все они находятся в одной иерархии интерфейсов), 
   так и специфические методы для популярных заголовков, таких как `Accept`, `Content-Type` и ряда других. 
   Дополнительно есть методы под разную семантику хедера `Authorization` - basic-авторизации и bearer token. В целом 
   класс `HttpHeaders` - наиболее популярный способ взаимодействия с заголовками, намного более удобный, чем 
   сервлетный API;
2. `T body`. `HttpEntity` - параметризованный класс. Позволяет принимать любой тип данных для тела. Он не слишком 
   популярен для ситуаций с передачей мультимедиа* или HTML, но при работе с API, предоставляющем и принимающем 
   текстовые данные в структурированном виде (JSON, XML и т.д.) - подобный механизм незаменим. При этом `HttpEntity` 
   все еще остается высокоуровневой моделью - он лишь хранит Java-объект, репрезентующий тело. `HttpEntity` не 
   занимается сериализацией такого объекта - это станет ответственностью определенной реализации 
   `HttpMessageConverter` в логике `DispatcherServlet` или конвертера внутри HTTP-клиента - в зависимости от того, кто 
   будет обрабатывать `HttpEntity`. 

> *Технически, никто не запретит передавать через `HttpEntity` мультимедиа - Spring имеет специальные классы, 
> позволяющие передавать данные любого формата. Наиболее известный - `Resource`. Также всегда остается возможность 
> использовать параметризацию в виде `String` или `byte[]`, если речь идет об относительно небольшом объеме.
> 
> Однако чаще всего `HttpEntity` для данных целей не используют. Как, впрочем, и Spring-приложение в целом.

Соответственно, конструкторы и методы `HttpEntity` завязаны на обработку описанных выше полей и удобного доступа к ним.

Конструкторы:

- `HttpEntity(T body)`. Принимает лишь тело без заголовков;
- `HttpEntity(MultiValueMap<String, String> headers)`. Принимает лишь заголовки без тела. Обратите внимание, параметр 
  заявлен как `MultiValueMap`, что позволяет передать как `HttpHeaders`, так и другую реализацию мапы при необходимости;
- `HttpEntity(T body, MultiValueMap<String, String> headers)`. Общий конструктор, позволяющий передать и заголовки, 
  и тело.

Стоит отметить, что `HttpEntity` пытается обеспечить имьютабельность класса, насколько это возможно. Теоретически это 
позволяет переиспользование объектов, их эффективное использование в многопоточной среде и прозрачность взаимодействия
в целом. Например, заголовки в этих целях оборачиваются в имьютабельную обертку, защищая от последующего изменения 
значений в пределах `HttpEntity`.

Однако на практике, никто не запрещает использовать в качестве тела изменяемый класс, в результате чего имьютабельность
`HttpEntity` будет неполной. `HttpHeaders` так же оставляет лазейки для изменения, то есть говорить о глубокой
неизменяемости в техническом плане не до конца верно. Однако практика использования позволяет улучшить несовершенство
API и добиться удобного, фактически неизменяемого и переиспользуемого объекта через использование имьютабельных DTO 
для тела и аккуртноую работу с заголовками.

Методы `HttpEntity`:

- `HttpHeaders getHeaders()`. Возвращает заголовки;
- `T getBody()`. Возвращает тело;
- `boolean hasBody()`. Проверяет наличие тела в объекте;
- Методы `Object`. `equals()`, `hashCode()` и `toString()` переопределены с использованием заголовков и тела. 
  Разумеется, будет работать корректно только в случае, если объект, записанный в тело, корректно реализует данные 
  методы.

Технически, `HttpEntity` можно использовать и для получения информации о запросе в контроллере, и в качестве 
возвращаемого значения контроллера. При отправке запросов, по аналогии, может использоваться и для передачи данных 
запроса, и для обработки данных ответа - все с поправкой на используемый HTTP-клиент. На практике этот класс до 
какого-то момента был популярен именно при исходящих запросах (и обработке ответа), но в последние годы он все менее 
популярен - подробнее затронем эту тему при знакомстве с HTTP-клиентами в Spring.

Также существует класс `RequestEntity` - наследник `HttpEntity`, заточенный на содержимое запроса: кроме тела и 
статуса он также содержит целевой URI и HTTP-метод. Однако он не слишком популярен, в силу чего мы не рассматриваем 
его в этой статье.

Ниже приведен пример использования класса в методе контроллера:

```java
@GetMapping("/hello-world/")
// В успешном сценарии будет автоматически использован код ответа 200.
// Если нужна более тонкая настройка - стоит использовать ResponseEntity.
public HttpEntity<HelloWorldDto> hello() {
    var headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_JSON);
    headers.set("My-Header-Name", "myHeaderValue");
    
    var body = new HelloWorldDto();
    
    return new HttpEntity<>(body, headers);
}
```

## `ResponseEntity` API

Как прямой наследник `ResponseEntity` предоставляет методы, доступные через `HttpEntity`, но также он дает ряд 
методов и несколько билдеров, позволяющих компактно описывать любой ответ через одну цепочку вызовов.

Начнем с нового поля: `HttpStatusCode status`. Оно представлено интерфейсом `HttpStatusCode`, который на практике 
чаще всего подразумевает имплементацию в виде енама `HttpStatus` - он содержит большое число статус-кодов для 
HTTP-ответов с указанием их стандартной семантики.

Конструкторы `ResponseEntity` также крутятся вокруг полей тела, заголовков и кода ответа. В том числе с возможностью 
передать код ответа числом (даже если такое число не представлено в `HttpStatus`). Таким образом, использование 
`ResponseEntity` оставляет за разработчиком максимальную гибкость по содержанию.

Однако на практике конструкторы `ResponseEntity` редко используются в явном виде - чаще используются статические
методы-фабрики или цепочки вызовов, пошагово собирающие конечный объект (реализация паттерна builder, он же - 
строитель).

Разберемся по порядку, от простого к сложному.

`ResponseEntity` предлагает два новых метода для доступа к содержимому:

- `HttpStatusCode getStatusCode()`. Возвращает объект кода ответа;
- `int getStatusCodeValue()`. Возвращает числовое значение кода ответа. В современных версиях Spring - `@Deprecated`.

Остальные методы так или иначе связаны с формированием самого объекта. Опять же, начнем с простых, которые 
фактически являются фабриками, возвращающими объект `ResponseEntity`:
- `ok(T body)`. Наиболее популярный метод, используемый для позитивных сценариев с кодом ответа 200. Принимает на 
  вход тело ответа;
- `of(Optional<T> body)`. Принимает `Optional` с телом ответа. Если `Optional` пуст - формирует ответ с кодом 404, 
  если нет - аналогичен `ok()`;
- `ofNullable(T body)`. Аналог `of()`, но принимает тело ответа без оберток. Если передан `null` - формируется 
  ответ 404, если объект есть - аналогичен `ok()`.

Остальные методы принимают на вход те или иные значения, но возвращают объекты билдеров, которые далее 
доконфигурируются уже через собственный API. Билдеры описаны двумя интерфейсами `HeadersBuilder` и `BodyBuilder`, 
причем `BodyBuilder` является наследником `HeadersBuilder`. Кратко пройдем по их API. Большинство методов актуально 
для достаточно узких сценариев и их лучше смотреть по ходу реализации конкретной задачи, ориентируясь на версию 
Spring, используемую в проекте. Поэтому вдаваться глубоко в детали нет смысла в подобной статье.

> Данная статья написана на базе Spring MVC 6.2.15. Соответственно, в более новых версиях могут быть добавлены новые 
> методы, а какие-то из текущих быть `@Deprecated` или даже удалены. В более старых версиях фреймворка часть методов 
> может отсутствовать.

`HeadersBuilder`:

- `header()`. Принимает на вход имя заголовка и массив (в виде varargs) его значений в строковом виде. Наиболее 
  общий метод для ситуации, когда необходимо добавить не самые популярные заголовки на уровне контроллера или, реже, 
  `@ControllerAdvice`-бина*. 
- `headers()`. Перегруженный метод, принимающий на вход объект `HttpHeaders` или же `Consumer<HttpHeaders>`. 
  Позволяет передать сразу все необходимые заголовки. Версия с `Consumer<HttpHeaders>` актуальна для дополнения 
  объекта `HttpHeaders`, уже помещенного в билдер. В дальнейшем мы довольно часто будем встречаться с ситуацией, когда
  добавление какой-то мета-информации или конфигурации происходит через лямбда-выражение;
- `allow()`. Конфигурирует заголовок `Allow`, принимает на вход varargs объектов `HttpMethod`. Семантически 
  `HttpMethod` близки к енаму, хоть и не являются им. Фактически эти значения указывают, какие методы доступны для 
  заданного ресурса (пути запроса);
- `eTag()`. Позволяет установить заголовок `ETag`. С официальной семантикой заголовка можно ознакомиться за пределами 
  статьи. Здесь не будем терять время, поскольку заголовок непопулярен в прикладной разработке;
- `lastModified()`. Конфигурация заголовка `Last-Modified`. Перегруженный метод, позволяющий передать временную 
  метку в том или ином формате (`ZonedDateTime`, `Instant`, `long`). Семантически указывает на дату последнего 
  обновления запрошенного ресурса**. На практике используется нечасто;
- `location()`. Управляет настройкой заголовка `Location`. Принимает на вход `URI` ресурса, который и запишется в 
  значение заголовка. Чаще всего используется в разрезе редиректов (коды ответа `3xx`). В современных приложениях с 
  клиентской частью на базе мощных JS-фреймворков перенаправления со стороны сервера - относительно редкая 
  функциональность, чаще роутинг происходит на стороне клиента. Перенаправление на внешние ресурсы часто выделяют в 
  обязанность какого-нибудь прокси-сервера, в результате чего на стороне бэкенд-приложения редиректы используются
  довольно редко. Однако в некоторых ситуациях `3xx` коды все еще актуальны, в том числе в некоторых сценариях 
  авторизации с использованием протоколов OAuth 2.0 или OIDC; 
- `cacheControl()`. Управляет заголовком `Cache-Control`, принимает на вход объект одноименного типа `CacheControl`. 
  Рекомендацию использовать данный хедер можно периодически встретить в различных мануалах по REST API, однако на 
  практике он зачастую игнорируется;
- `varyBy()`. Настраивает заголовок `Vary`, принимает на вход имена заголовков запроса. В семантику углубляться не 
  будем - она не самая очевидная и актуальна далеко не для всех систем. Опять же, используется крайне редко;
- `build()`. Финальный метод билдера, завершающий конфигурацию объекта и возвращающий получившийся объект 
  `ResponseEntity`.

> *Обычно добавление заголовков, особенно не самых типичных, является частью сквозной функциональности. И чаще всего 
> такие заголовки добавляются на более низком уровне, нежели контроллер - через сервлетные фильтры или иные 
> механизмы (например, предоставляемые Spring Security).
> 
> **В современных реалиях под ресурсом зачастую будет пониматься объект, построенный на основе записи в БД или 
> иным образом хранящийся в системе. У новичков данный термин часто вызывает ступор из-за его широкого использования 
> в совершенно разных контекстах. В принципе, ресурсом может быть и HTML-страница (что наиболее очевидно при 
> оперировании таким термином в ключе HTTP-коммуникации) или что-либо еще, вплоть до файла. Однако если мы говорим о 
> REST API или концептуально схожих форматах, речь почти всегда будет идти именно о представлении какого-то 
> внутреннего объекта, чаще всего доступного по ID. 

`BodyBuilder` добавляет буквально несколько новых методов, так или иначе касающихся тела ответа:

- `contentLength()`. Настраивает `Content-Length`, принимает на вход размер тела ответа в байтах. Для типичных 
  ситуаций с предоставлением текстовых или бинарных данных редко заполняется на уровне приложения - зачастую этим 
  управляет контейнер сервлетов, как, впрочем, и сам Spring MVC через `HttpMessageConverter` и иные механизмы;
- `contentType()`. Управляет `Content-Type`, на вход принимает `MediaType`. Опять же, редко управляется приложением. 
  Чаще всего Spring MVC устанавливает это значение самостоятельно, опираясь на то, во что он фактически сериализует
  ответ. Также значение можно установить декларативно через `@RequestMapping.produces()`;
- `body()`. Принимает на вход тело ответа, возвращает получившийся объект `ResponseEntity`.

По итогу все выглядит довольно просто, нам осталось лишь рассмотреть набор методов `ResponseEntity`, которые 
возвращают описанные выше билдеры.

Сами по себе такие методы управляют установкой кода ответа, а возвращаемый ими билдер опирается на то, ожидается ли 
тело ответа при выбранном коде в соответствии с семантикой HTTP. Сами методы, которые мы рассмотрим ниже, 
довольно редко используются на уровне методов контроллеров (по крайней мере, `@RequestMapping`-методов) - 
большинство из них обозначают `4xx` или `5xx` коды, которые обозначают ошибки. Чаще их можно увидеть в 
`@ExceptionHandler`-методах - внутри классов, помеченных `@ControllerAdvice`, или внутри самих контроллеров. Чуть 
детальнее рассмотрим этот вопрос при углублении в тему обработки ошибок в Spring MVC.

Методы, возвращающие `BodyBuilder`:

- `status()`. Перегруженный метод, принимающий на вход код ответа в виде `HttpStatusCode` или числового эквивалента. 
  Наиболее общий из всех, поскольку позволяет указать абсолютно любой код;
- `ok()`. Устанавливает 200 код ответа. Актуален в тех случаях, когда кроме тела ответа необходимо установить и 
  какие-то заголовки - в остальных случаях гораздо проще использовать фабричный `ok(T body)`, сразу возвращающий 
  `ResponseEntity`;
- `created()`. Устанавливает 201 код ответа, принимает на вход URI, записываемый в заголовок `Location`. Фактически 
  описывает второй сценарий использования `Location` - предоставление ссылки на созданный ресурс. В рекомендациях 
  для REST API регулярно советуют использовать 201 код для успешных ответов на запросы, создающие ресурс (в 
  классическом REST CRUD для этого обычно используется `POST`-запрос). Но на практике эта рекомендация часто
  игнорируется;
- `accepted()`. 202 код ответа. Семантика HTTP регламентирует использование этого кода в ситуациях, когда запрос 
  принят, но обработка фактически не завершена - обычно в контексте длинных асинхронных задач, когда реальный результат 
  будет получен каким-то иным способом, а задача текущего метода - лишь инициировать процесс. На практике 
  используется нечасто, даже в сценариях, близких к заявленной семантике;
- `badRequest()`. 400 код ответа. Обозначает некорректное содержимое запроса - например, переданные данные не прошли 
  валидацию. Довольно популярный код, в теле ответа чаще всего будет описание произошедшей ошибки (классический 
  сценарий - какие поля тела запроса не прошли валидацию и почему);
- `unprocessableEntity()`. 422 код ответа. Не популярен, в семантику углубляться не будем;
- `internalServerError()`. 500 код ответа. Наиболее популярный из всех статусов, связанных с ошибками. 
  Фактически означает ошибку сервера, но при отсутствии качественного механизма обработки ошибок сюда 
  могут попасть и классические клиентские ошибки (некорректный запрос, отсутствие авторизации, 
  отсутствие доступа и так далее). Используется как дефолтный код ответа при ошибке на уровне контейнера сервлетов.
  Тело ответа обычно содержит описание ошибки, может также содержать stack trace исключения* или какие-то иные детали.

> *Stack trace - относительно частая история. Во-первых, это стандартное поведение для некоторых контейнеров 
> сервлетов (включая Tomcat), во-вторых, это облегчает разработчикам локализацию ошибки, особенно во внутреннем 
> тестировании.
> 
> Однако для коммерческой разработки это нежелательная практика. Она предоставляет клиенту детали реализации системы,
> которые не имеют для него значения. В отдельных случаях это может быть использовано злоумышленниками. Кроме того, 
> это банально увеличивает объем данных, передаваемых по сети.
> 
> Более корректным будет формализация содержимого в ответах таким образом, чтобы это было информативно и для 
> клиентского приложения (ведь ему зачастую нужно как-то описать ошибку пользователю), и для разработчика (коды 
> ошибки, идентификация трейса для поиска информации в логах приложения и так далее). Однако это отдельная большая 
> тема, которую мы частично рассмотрим в следующих статьях.

Методы, возвращающие `HeadersBuilder`:

- `of(ProblemDetail body)`. Позволяет установить любой статус ответа, но создан для негативных сценариев с `4xx` или 
  `5xx` кодами. Сам код ответа указывается внутри объекта `ProblemDetail`, там же можно описать и детали ошибки, в том 
  числе с передачей набора дополнительных атрибутов, для обработки на стороне клиента. `of()` иногда используется в 
  `@ExceptionHandler`-методах, но чаще остается обделен вниманием, из-за чего мы не углубляемся в API данного класса 
  в этой статье;
- `noContent()`. 204 код ответа. Регламентирован для ситуаций, когда запрос обработан успешно, но сервер не должен 
  возвращать какую-либо полезную нагрузку. Используется нечасто, хотя сама ситуация с пустым телом ответа - не редкость;
- `notFound()`. 404 код ответа - ресурс не найден. Весьма популярный статус, семантика очевидна.

С точки зрения практики применения `ResponseEntity` крайне популярен: чаще всего он используется в ответах 
методов контроллеров, также используется при обработке ошибок через `@ExceptionHandler`-методы. Наконец, данный тип 
используется для обработки ответа на исходящий запрос - здесь он выступает как read-only объект и этот вариант мы 
детальнее рассмотрим при знакомстве с HTTP-клиентами.

Для закрепления рассмотрим несколько примеров с формированием объекта `ResponseEntity` через тот или иной API. Часть 
из них мы закрепим повторно в статье, посвященной обработке ошибок.

```java
// 1. Наиболее типичный сценарий: 200 код ответа и тело
ResponseEntity.ok(new HelloWorldDto());

// 2. Успешный ответ, тело не требуется. Обычно метод контроллера будет описывать возвращаемое значение как 
// ResponseEntity<Void> или ResponseEntity<?>
ResponseEntity.noContent()
    .build();

// 3. Успешный ответ с телом и установкой кастомного заголовка
ResponseEntity.ok()
    .header("My-Header-Name", "myHeaderValue")
    .body(new HelloWorldDto());

// 4. Аналогично предыдущему, но также с явно установленным Content-Type
ResponseEntity.ok()
    .contentType(MediaType.APPLICATION_JSON)
    .header("My-Header-Name", "myHeaderValue")
    .body(new HelloWorldDto());

// 5. 500 код и описание ошибки в теле ответа. Здесь пример упрощен, на практике ErrorDto обычно содержит описание 
// ошибки, иногда - внутренний код ошибки в системе, trace ID или иные детали, которые помогут либо корректно 
// обработать ошибку на стороне клиента, либо помогут в локализации и исправлении ошибки службе поддержки и 
// разработчикам
ResponseEntity.internalServerError()
    .body(new ErrorDto());
```

Также попробуем переделать на использование `ResponseEntity` описанный выше пример с `HttpEntity`:

```java
@GetMapping("/hello-world/")
public ResponseEntity<HelloWorldDto> hello() {
    var headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_JSON);
    headers.set("My-Header-Name", "myHeaderValue");

    var body = new HelloWorldDto();

    return new ResponseEntity<>(body, headers, HttpStatus.OK);
}
```

Несложно заметить, что аналогичное поведение у примера 4, описанного выше. Но в данном случае преобразование более 
наглядное и фактически не меняется ничего, кроме имени класса и явного указания кода ответа.
