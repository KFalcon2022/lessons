# Конфигурация Spring MVC

Сегодня мы запустим первое приложение на основе Spring MVC. Это будет почти полный аналог
[первого сервлетного приложения](https://github.com/KFalcon2022/servlet-sample), разобранного в соответствующем разделе.

Задача текущей статьи - продемонстрировать наиболее базовые шаги конфигурирования и запуска: какие зависимости нужны,
какая Spring-конфигурация требуется, как собрать приложение.

Многие вещи будут нам знакомы, ведь фактически мы все еще будем работать с сервлетным приложением, лишь нарастив на 
него несколько слоев абстракции. Параллельно начнем знакомиться с терминами и подходами, вокруг которых построены 
данные абстракции.

Поскольку мы вновь возвращаемся к веб-приложениям - потребуется предустановленный Tomcat или иной контейнер сервлетов.

## Зависимости

Первое, что нужно для создания Spring MVC приложения - подключить соответствующие библиотеки и плагины. Ниже 
приведен пример для Gradle:

```groovy
plugins {
    id 'java'
    id 'war' // Как и для обычных сервлетных приложений, требуется упаковка в WAR-архив
}

// ... - Настройка репозиториев, версии Java и другие стандартные шаги

dependencies {
    // Spring MVC по умолчанию включает в себя Spring Core и другие стандартные блоки, включая AOP (без
    // AspectJ-зависемостей). Соответственно, нет необходимости указывать эти зависимости явно -
    // они будут поставлены транзитивно.
    implementation 'org.springframework:spring-webmvc:6.2.15'

    // Spring MVC не поставляет Servlet API, но использует его в собственной логике. Поэтому зависимость потребуется 
    // добавить мануально, даже если в коде приложения не будет явных обращений к ней. 
    compileOnly 'jakarta.servlet:jakarta.servlet-api:6.1.0'
}
```

Здесь все довольно стандартно - у нас появилась одна новая зависимость, в остальном нет каких-то значимых отличий от
конфигурации сервлетного приложения.

## Java-конфигурация

Здесь нам потребуется сделать сразу несколько вещей:

1. Настроить component scan. Это типичная задача для вообще любого Spring-приложения;
2. Сконфигурировать непосредственно Spring MVC. В общем случае этот шаг может затрагивать огромное число настроек, 
   от безопасности (скажем, [CORS](https://ru.wikipedia.org/wiki/Cross-origin_resource_sharing)) и поддерживаемых 
   типов контента (JSON, XML и т.д.) до настроек валидации, interceptor'ов (перехватчики запроса, семантически 
   напоминающие сервлетные фильтры) и многого другого. Благо, на первых шагах об этом можно не думать, доверившись
   стандартным настройкам Spring MVC;
3. Сконфигурировать `DispatcherServlet` - звучит страшно, но концептуально это аналог конфигурации сервлетного контекста
   через `web.xml`, только в реалиях абстракций, добавленных Spring MVC.

С точки зрения кода это выражается в двух небольших классах, если нам не требуется какой-то сложной настройки::

```java
@Configuration
// Обычный @ComponentScan, с ним мы знакомы еще по Spring Core. Шаг 1 в списке выше.
@ComponentScan("com.walking.springmvc")
// Конфигурируем Spring MVC. Для нас это выглядит как одна enable-аннотация, поскольку мы никак не меняем 
// стандартные настройки. В более продвинутом варианте WebConfig был бы наследником WebMvcConfigurer и переопределял 
// необходимые методы.
@EnableWebMvc
public class WebConfig {
}

// Класс, описывающий настройки DispatcherServlet'а. Подробнее - ниже
public class WebAppInitializer extends AbstractAnnotationConfigDispatcherServletInitializer {
    @Override
    protected Class<?>[] getRootConfigClasses() {
        return null;
    }

    @Override
    protected Class<?>[] getServletConfigClasses() {
        return new Class<?>[]{WebConfig.class};
    }

    @Override
    protected String[] getServletMappings() {
        return new String[]{"/"};
    }
}
```

Идея Spring MVC строится вокруг dispatcher servlet - сервлета, отвечающего за диспетчеризацию запросов. Иными 
словами, Spring подумал и решил, что давать контейнеру сервлетов самому заниматься маппингом запросов - это как-то 
слишком, да и вообще не умеют они это делать эффективно*. Так или иначе, весь Spring MVC строится на том, что с точки 
зрения контейнера все запросы обрабатываются одним сервлетом (обычно с базовым путем `/`), а вся дальнейшая 
диспетчеризация до фактического обработчика - ответственность этого `DispatcherServlet`. Стоит отдать должное, это дает 
довольно много плюсов, делая код более гибким и интуитивно-понятным.

> *По мнению разработчиков Spring, разумеется.

Таким образом dispatcher servlet фактически берет на себя ту ответственность, которая лежала на контейнерах 
сервлетов и настраивалась, в частности, через `web.xml`. Сама диспетчеризация запросов никуда не делась, она лишь 
перенеслась на более высокий уровень.

Однако между голым сервлетным API и современными приложениями в связке Spring MVC + Spring Boot было несколько 
промежуточных эволюционных звеньев, следы которых и сейчас можно найти в исходниках. Стоит выделить два из них:

1. Несколько Spring-контекстов, объединяемых в иерархию. Фактически это было ответом Spring на подход, основанный на 
   размещении нескольких сервлетных приложений с собственными сервлетными контекстами в одном контейнере сервлетов. 
   Иерархия `ApplicationContext` позволяла как объединить общие элементы в одном родительском контексте, так и 
   инкапсулировать специфику отдельных приложений в дочерних контекстах. Современные подходы обычно предполагают, 
   что один контейнер сервлетов обслуживает одно приложение. Как апофеоз этой модели Spring Boot в принципе 
   помещает контейнер сервлетов внутрь собственного модуля (стартера), то есть приложение запускает условный Tomcat 
   внутри себя. Пока мы знакомимся с "голым" Spring MVC, нам придется работать с отдельно установленным Tomcat, что 
   требует дополнительной конфигурации;
2. Настройка `DispatcherServlet`. Собственно, это то, что сейчас делает `WebAppInitializer` из примера выше. 
   Теоретически, в приложении может быть несколько `DispatcherServlet`-ов, слушающий разные пути (или шаблоны путей).
   Тогда конфигов, аналогичных `WebAppInitializer` тоже будет несколько - по одному на сервлет. Если попытаться 
   спроецировать зону ответственности классов-initializer'ов на сервлетные приложения - каждый такой класс настраивает
   собственный servlet context.

Теперь чуть подробнее о методах, которые мы видим в `WebAppInitializer`:

- `getRootConfigClasses()`. Классы конфигурации корневого контекста. Это конфиги для родительского контекста, 
  упоминаемого выше. Поскольку у нас нет иерархии контекстов, нет и родительского. Можно смело возвращать `null`, 
  это актуально для 99% свежих проектов. Если бы действие происходило лет 15-20 назад - мы бы могли, скажем, 
  захотеть разместить на одном сервере и приложение, и его админку. И тогда бы часть сервисов и домен могли быть 
  общими (и конфигурироваться через родительский/корневой контекст), а остальное относилось бы к отдельным дочерним 
  контекстам - у приложения свое, у админки - свое. Благо, этот архитектурный подход канул в Лету;
- `getServletConfigClasses()`. Классы конфигурации для текущего dispatcher servlet'а/servlet context'а. Поскольку у 
  нас нет каких-то значимых переопределений - достаточно указать на один класс конфигурации - `WebConfig`. Массив 
  может потребоваться, если различные элементы web-конфигурации настраиваются отдельными бинами - мы рассмотрим, что 
  это за элементы при более глубоком погружении в работу dispatcher servlet. Однако даже в современном приложении, 
  захоти мы написать его без Spring Boot обычно будет достаточно одного аккумулирующего класса;
- `getServletMappings()`. Единственное, что на самом деле важно для нас. Описывает все пути (или паттерны путей), 
  которые должен обрабатывать именно этот dispatcher servlet. В нашем случае все просто - `/` указывает, что все 
  запросы идут через данный dispatcher servlet*. В примере с приложением и админкой мы могли бы, скажем, один 
  dispatcher servlet настроить на паттерн `/app/*` - обработку запросов приложения, а второй (в отдельном 
  классе-initializer'е) - на `/admin/*` и запросы к админке создавать строго с таким префиксом.

> На данном этапе стоит раскрыть разницу между паттернами `/` и `/*`.
> 
> Первый вариант направляет в dispatcher servlet все запросы, которые не маппятся на другие сервлеты (вряд ли
> актуально здесь, но имеет значение в классических сервлетных приложениях) или на статические ресурсы (в принципе,
> может быть актуально даже со Spring MVC).
> 
> `/*` - более строгое правило, означающее буквально все запросы.
> 
> В учебных задачах разница едва ли заметна, но в промышленной разработке, особенно на legacy-проектах, может иметь 
> значение. Благо, повсеместно в обучающих материалах демонстрируется именно `/`.

Хорошая новость заключается в том, что `AbstractAnnotationConfigDispatcherServletInitializer` нас интересует очень 
поверхностно и лишь в разрезе Spring MVC. После знакомства со Spring Boot необходимость в нем отпадет. Там это 
преднастройку возьмет на себя сам фреймворк. При желании, сконфигурировать что-то экзотическое можно и там, но это 
точно не стоит рассматривать в пределах курса.

## Контроллер

Остается самая приятная часть - бизнес-логика.

Если в сервлетных приложениях для описания обработки запроса мы используем классы-сервлеты, то в Spring MVC это 
ответственность специального вида Spring-компонентов - контроллеров. Они имеют иную логическую структуру, что 
позволяет гораздо гибче подходить к вопросу объединения запросов по разным путям в одном классе, нежели при 
использовании Servlet API:

```java
// Маркерная аннотация вроде Service или Repository. Используется именно для обозначения обработчиков запросов
@Controller
// @RequestMapping над классом указывает общий префикс, который обрабатывает именно этот класс-контроллер. Можно 
// указывать несколько значений, что бывает полезно для миграции путей с сохранением обратной совместимости без 
// дублирования кода 
@RequestMapping("/hello")
public class HelloWorldController {
    // @ResponseBody указывает, что мы хотим вернуть не какой-то статический ресурс по заданному имени, а именно то, 
    // что возвращаем из метода. В данном случае - строку, но тут может быть и некий объект, сериализуемый в JSON или 
    // иной формат. Абсолютное большинство современных Spring-приложений используют эту аннотацию в неявном виде. Но 
    // это тема следующих статей
    @ResponseBody
    // @GetMapping - дочерняя аннотация относительно @RequestMapping. Во-первых, в данном случае зашивается 
    // используемый метод HTTP (существуют также и @PostMapping, @PutMapping и т.д.). Во-вторых, расположение над 
    // методом показывает, что именно этот метод обрабатывает именно этот путь
    @GetMapping("hello-world")
    // Метод может работать не с ServletRequest и ServletResponse, а с типами, более близкими к бизнес-логике. В 
    // ServletResponse наша строка запишется уже силами Spring MVC в виде тела ответа
    public String hello() {
        return "Hello, World!";
    }
}
```

По итогу сейчас наше приложение готово обрабатывать GET-запросы по пути `${dispatcherUrlPrefix}/hello/hello-world`. 
Или, с учетом нашей настройки dispatcher servlet - просто `/hello/hello-world`. Это в целом схоже с тем, как 
работало первое сервлетное приложение (ссылка в начале статьи). Единственное различие - там `/hello` было оформлено как 
context path, что соответствовало реалиям применения той технологии. Здесь же context path, по сути - `/`, а 
префикс `hello/` - уже внутренний префикс.

## Развертывание

Итак, приложение написано, как же его запустить?

Со сборкой все также, как и для сервлетных приложений - `gradle build` или, более специализировано, `gradle war` 
упакует все в WAR-архив, который в дальнейшем требуется подложить в Tomcat или иной контейнер сервлетов.

Сделать это можно как через IDEA (`Tomcat Server` -> `Local` в конфигурациях запуска), или же вручную, поместив WAR 
внутрь `${tomcatHome}/webapps`.

Поскольку мы переходим к современному подходу с одним приложением на сервер - рекомендую или при настройке 
конфигурации в IDEA указать в `Deployment` -> `Application Context` значение `/`, или же сам WAR-файл создавать с 
именем `ROOT.war` (можно настроить в `build.gradle`) и подкладывать его в `${tomcatHome}/webapps`, предварительно 
удалив стандартную ROOT-директорию.

Посмотреть код приложения целиком можно здесь: [ссылка](https://github.com/KFalcon2022/spring-mvc-sample).
